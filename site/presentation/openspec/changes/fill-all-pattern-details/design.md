## Context

Инфраструктура для `detail` готова (change `pattern-detail-fullscreen`). `PatternItem` имеет опциональное поле `detail?: PatternDetail` с полями `problem`, `solution`, `quote`, `worksWhen`, `failsWhen`, `visual`. Источник контента — `engineering-patterns.md` (~370 строк, 10 разделов, каждый паттерн описан в формате «Суть / Где встречается / Когда работает / Когда ломается»).

## Goals / Non-Goals

**Goals:**
- Заполнить `detail` для всех паттернов по материалам `engineering-patterns.md`
- Единообразный стиль: problem краткий (2-3 предложения), solution конкретный, quote прямая цитата из источника где есть, visual ASCII

**Non-Goals:**
- Изменение компонентов или CSS
- SVG/React визуалы (ASCII достаточно для пилота)
- Добавление новых паттернов

## Decisions

**1. Один task per категория**

10 категорий → 10 задач. Каждая задача — самодостаточная: прочитать раздел в `engineering-patterns.md`, заполнить `detail` для всех паттернов категории.

**2. ASCII visual — не для каждого паттерна**

Не все паттерны выигрывают от ASCII-диаграммы. Правило: visual только если есть явный поток данных, архитектурная схема или сравнение before/after. Процессные и культурные паттерны (Fresh session, Прыжок веры) — без visual.

**3. Источник цитат — строго из `engineering-patterns.md`**

Цитаты (`quote`) берём только из поля «Где встречается» в источнике, где есть прямая речь с атрибуцией. Не синтезируем.

**4. `failsWhen` — опциональный**

Для антипаттернов раздел `failsWhen` не нужен — сам паттерн уже описывает проблему. Поле пропускаем.

## Risks / Trade-offs

- [Неравномерность источника] → Некоторые паттерны описаны кратко. Для них `detail` будет минимальным (problem + solution), это нормально — fallback всё равно лучше пустоты
- [Большой объём patterns.ts] → Файл вырастет до ~900-1000 строк. Приемлемо, TypeScript компилирует без проблем
