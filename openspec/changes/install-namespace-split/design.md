## Context

awesome-claude — набор скиллов и правил для Claude Code. Сценарий: пользователь говорит "установи из <URL>" — Claude должен автономно разобраться, что там есть, и предложить план. Без автономного исследования пользователь вынужден знать структуру репо заранее.

Текущая структура: `skills/<ns>/` — по неймспейсам, `rules/` — плоская, не разбита. Нужно добавить `rules/<ns>/` и корневые `rules/` (универсальные).

Принцип claude-way: Claude Code — единственный интерфейс.

## Goals / Non-Goals

**Goals:**

- D1: `dev:install <url>` запускает pipeline: **clone → explore → propose → confirm → install** — в этом порядке, всегда
- D2: **explore** — динамически обнаруживает неймспейсы: читает `ls skills/`, для каждого `<ns>` собирает список скиллов (из `skills/<ns>/*/skill.md` frontmatter `name:`) и список правил (из `rules/<ns>/`)
- D3: **propose** — Claude формирует текстовое описание каждого неймспейса ("dev: 10 скиллов — tdd, fix-bug, ...; правила: git.md, break-stop.md"), затем AskUserQuestion с чекбоксами по неймспейсам
- D4: **install** — для каждого выбранного `<ns>`: `skills/<ns>/` → `.claude/skills/<ns>/`, `commands/<ns>/` → `.claude/commands/<ns>/`, `rules/<ns>/` → `.claude/rules/<ns>/`; корневые `rules/*.md` + `CLAUDE.md` устанавливаются всегда
- D5: если `rules/<ns>/` не существует для неймспейса — пропускается без ошибки (rules опциональны для неймспейса)
- D6: если неймспейс уже установлен — в propose указывается "(уже установлен, будет обновлён)"

**Non-Goals:**

- Анализ содержимого скиллов глубже метаданных frontmatter
- Управление зависимостями между неймспейсами
- Удаление установленных компонентов

## Decisions

**D1 — pipeline как жёсткая последовательность**

Explore и propose — обязательные шаги, не optional. Нельзя вызвать install минуя propose. Гарантия: Claude всегда понимает что устанавливает, пользователь всегда видит предложение перед изменениями.

**D2 — explore динамический, не хардкодит неймспейсы**

Список неймспейсов = `ls skills/` в клонированном репо. Скилл не знает заранее что там есть — это позволяет `dev:install` работать с любым репо той же структуры.

**D3 — rules/<ns>/ как опциональная директория**

Не каждый неймспейс обязан иметь свои правила. `rules/<ns>/` создаётся только при наличии namespace-специфичных правил. Корневые `rules/*.md` — универсальные (claude-way, break-stop, git, meta-rules) — всегда идут в `.claude/rules/`.

**D4 — реструктуризация существующих rules**

Текущие rules распределяются по принципу: что относится к конкретному неймспейсу → переезжает в `rules/<ns>/`, что универсально → остаётся в `rules/`.

Примерное распределение:
- `rules/` (универсальные): `claude-way.md`, `break-stop.md`, `git.md`, `meta-rules.md`
- `rules/dev/`: `makefile.md`, `monorepo-structure.md`, `frontend-design.md`, `frontend-testing.md`, `ui-library.md`, `arch/`
- `rules/skill/`: `skill-tdd-coverage.md`

**D5 — clone в tmp с переиспользованием**

Если `/tmp/awesome-claude-install` существует и remote совпадает с url — переиспользуется. Иначе — удаляется и клонируется заново.

## Risks / Trade-offs

- [Risk] Репо не содержит `skills/<ns>/` структуры → Mitigation: explore сообщает "структура не распознана" и останавливается
- [Risk] Пользователь снимает все галочки → Mitigation: install не запускается, сообщение "ничего не выбрано"
- [Risk] Реструктуризация rules ломает существующие установки → Mitigation: `.claude/rules/` у пользователей обновляется при следующем `dev:install`
