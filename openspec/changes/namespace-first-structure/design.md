## Context

Текущая организация репозитория использует тип артефакта как первичную ось:

```
skills/dev/
skills/sdd/
commands/dev/
commands/sdd/
rules/         ← плоско, монолит
docs/          ← плоско, только при полной установке
```

Это приводит к двум проблемам:

1. **Нельзя установить namespace целиком**. `install.sh dev` ставит `skills/dev/` и `commands/dev/`, но не ставит `rules/` и `docs/` — они общие. Чтобы получить полный пакет dev, нужна вся репа целиком.
2. **Установка зависит от знания путей**. Любой инструмент (включая Claude в auto-deps) должен заранее знать что `skills` идёт в `.claude/skills/<ns>/`, а `commands` — в `.claude/commands/<ns>/`.

Связан с `skill-auto-deps`: там Claude должен на ходу разбираться как ставить namespace. Самоописываемая структура делает это детерминированным.

## Goals / Non-Goals

**Goals:**
- Namespace становится самодостаточным пакетом: одна директория `<ns>/` содержит всё, что нужно для namespace
- Установка namespace = копирование одной директории по предсказуемой схеме
- `shared/` для артефактов, нужных нескольким namespace-ам
- Структура самоописываемая: Claude может установить namespace, изучив директорию, без знания `install.sh`

**Non-Goals:**
- Не ломать совместимость с Claude Code: финальная структура в `.claude/` остаётся прежней (`.claude/skills/<ns>/`, `.claude/commands/<ns>/` и т.д.) — меняется только структура **репозитория-источника**
- Не вводить symlink'и или другие нестандартные механизмы — простое копирование
- Не разделять crosscutting артефакты — для них есть `shared/`

## Decisions

**Структура репозитория: namespace на верхнем уровне**

```
dev/
  skills/
  commands/
  rules/
  docs/
sdd/
  skills/
  commands/
  rules/
  docs/
research/
  ...
shared/
  rules/
  docs/
```

`shared/` устанавливается всегда автоматически. Альтернатива: явная зависимость через `requires:` на уровне namespace — отклонено как избыточное усложнение для общих ресурсов; `shared/` концептуально проще.

**Маппинг при установке детерминирован**

```
<ns>/skills/   → .claude/skills/<ns>/
<ns>/commands/ → .claude/commands/<ns>/
<ns>/rules/    → .claude/rules/<ns>/
<ns>/docs/     → .claude/docs/<ns>/
shared/rules/  → .claude/rules/shared/
shared/docs/   → .claude/docs/shared/
```

Никаких исключений и manifest-файлов — структура говорит сама за себя.

**Дублирование вместо переиспользования для небольших общих частей**

Если две-три namespace используют один rule — лучше продублировать, чем создавать сложную систему ссылок. `shared/` только для действительно крупных или универсальных вещей (например, общая документация по skill design, общие архитектурные правила).

Альтернатива: `requires:` между namespace-ами. Отклонено — добавляет неявные зависимости и усложняет установку, выгода неочевидна.

**Claude Code продолжает читать `.claude/skills/<ns>/`**

Финальная раскладка в `.claude/` не меняется — только структура репозитория-источника. Это значит:

- Claude Code и сами скиллы не нужно адаптировать
- Меняется только `install.sh` и логика разворачивания

## Dependencies / Follow-ups

Change `post-release-version-bump` принял ряд решений, которые нужно пересмотреть после реализации `namespace-first-structure`:

1. **`.manifest` с полем `namespaces:`** — в новой структуре становится избыточным. Самоописываемая директория `<ns>/` сама говорит что копировать. При реализации `namespace-first-structure` — убрать `namespaces:` из `.manifest` или удалить `.manifest` целиком.

2. **`bump-namespace.sh` пути** — скрипт клонирует `skills/<ns>/` из remote. После переструктурирования нужно клонировать `<ns>/` и маппить подпапки. Обновить пути в скрипте.

3. **`claude-entry-point` через `index.md`** — проверить совместимость с `shared/docs/`. Возможно `index.md` логичнее лежит в `shared/docs/` вместо `.claude/awesome-claude/`.

## Risks / Trade-offs

[Дублирование расходится со временем] → если правило дублировано в `dev/` и `sdd/`, изменение в одном не дойдёт до другого. Митигация: документировать в `docs/REPO_ORGANIZATION.md` критерий "когда `shared/`, когда дублирование": >2 namespace-ов или критичная общая логика → `shared/`; иначе дублируем.

[Большая миграция файлов] → перенос `skills/dev/` → `dev/skills/` затрагивает все namespace. Митигация: сделать одной командой через git mv в одном PR. Зеркало `.claude/` обновляется тем же скриптом.

[`shared/` может стать свалкой] → соблазн положить туда всё подряд. Митигация: правило "shared только если используется ≥2 namespace-ами или критично универсально"; review при каждом добавлении в shared.

[Несовместимость с предыдущими установками] → у пользователей `.claude/` был сформирован старой версией install.sh. Митигация: новая install.sh идемпотентна — затирает существующий namespace при переустановке, поэтому достаточно `bash install.sh <ns>` ещё раз.
