## Why

Когда пользователь вызывает скилл внутри неймспейса (например, `/sdd:propose`), Claude не знает, что следует вызвать дальше — нет машиночитаемого объявления последовательности. Поле `workflow_step: N` существует, но не раскрывает полную цепочку и не указывает, какие скиллы идут до и после.

Дополнительно: поле `description:` в frontmatter сейчас — краткий однопараграфный текст. Этого недостаточно для Claude, чтобы точно определить условия вызова, предусловия, ожидаемые входы и выходы, отличия от соседних скиллов по неймспейсу.

Оба пробела приводят к тому, что Claude либо запрашивает уточнение у пользователя, либо называет следующий шаг неверно.

## What Changes

- Ввести стандартный расширенный формат описания скилла в frontmatter: разделы `triggers`, `preconditions`, `inputs`, `outputs`, `next_skill`
- Ввести поле `namespace_sequence` в frontmatter каждого скилла неймспейса: упорядоченный список всех скиллов неймспейса с кратким описанием каждого
- Обновить все скиллы `sdd/` неймспейса: `sdd:explore`, `sdd:propose`, `sdd:contradiction`, `sdd:apply`, `sdd:archive` — добавить `namespace_sequence` и расширенный `description`
- Обновить `docs/SKILL_DESIGN.md` — описать новый формат frontmatter и правила заполнения
- Обновить зеркала в `.claude/skills/sdd/`

## Capabilities

### New Capabilities

- `skill-rich-description`: стандартный расширенный формат frontmatter скилла: `triggers`, `preconditions`, `inputs`, `outputs`, `next_skill`

## Impact

- Claude при выполнении любого скилла сможет прочитать `namespace_sequence` и однозначно подсказать пользователю следующий шаг
- Поле `next_skill` позволяет Claude завершать скилл с явной рекомендацией: «Следующий шаг — `/sdd:apply`»
- Расширенный `description` снижает число ошибочных вызовов: пользователь не получает "не тот" скилл при похожих запросах
- Изменения обратно совместимы: новые поля опциональны для скиллов не из sdd/ неймспейса

See `.sdd.yaml` for capability declarations.
