## Why

Сейчас вывод Claude после ключевых workflow-скиллов (`sdd:apply`, `sdd:contradiction`, `sdd:archive`) — сплошной AI-текст: технические факты, прозаическое описание, вопросы к пользователю и сторонние замечания смешаны в одном потоке. Пользователь не может за один взгляд понять «что точно сделано», «о чём со мной говорят» и «на какие вопросы я обязан ответить». По итогам `sdd:apply` нет явного списка реализованных фич; по итогам `sdd:contradiction` — структурированный отчёт есть, но в нём не выделено, что именно требует ответа пользователя; по итогам `sdd:archive` нет явного перечня заархивированных capabilities.

См. `.sdd.yaml` для capability-декларации.

## What Changes

- Ввести обязательный **фиксированный порядок** user-facing вывода для трёх скиллов (`sdd:apply`, `sdd:contradiction`, `sdd:archive`). Информационное и не требующее ответа — выше; то, на что пользователь обязан реагировать, — в самом конце. Полный канонический порядок:
  1. **Технические статусы** — факты: пути файлов, результаты тестов, exit codes, состояние артефактов. Без прозы.
  2. **Описание** — короткая прозаическая сводка того, что произошло.
  3. **Итоги** — per-skill блок результата (`## Реализованные фичи` / `## Найденные противоречия` / `## Архивированные артефакты`).
  4. **Решено самостоятельно** (опц.) — вопросы и развилки, которые Claude закрыл автономно. Формат: «вопрос → решение». Блок опускается, если решений не было. Не требует реакции пользователя.
  5. **Прочее** (опц.) — риторические замечания, follow-up идеи. Опускается, если пусто. Не требует реакции пользователя.
  6. **Вопросы к пользователю** — **последний** блок отчёта. Содержит только то, что выходит за компетенцию Claude (доменное знание, бизнес-приоритеты, личные предпочтения). Если таких вопросов нет — блок состоит ровно из одной строки CTA (см. ниже).
- **Call-to-action — обязателен**. Последняя строка блока «Вопросы к пользователю» — всегда CTA. По умолчанию это `1. Продолжаем по флоу? (да / нет)`. Если есть user-only вопросы, они идут до CTA в том же блоке; CTA — всегда финальная строка отчёта.
- Per-skill содержимое блока «Итоги»:
  - `sdd:apply` → `## Реализованные фичи` — список capabilities из `.sdd.yaml.creates` со статусом `done`/`partial`.
  - `sdd:contradiction` → `## Найденные противоречия` — компактный summary найденных hard issues и warnings (полный детализированный отчёт скилла остаётся как есть и идёт внутри блока «Технические статусы»).
  - `sdd:archive` → `## Архивированные артефакты` — список заархивированных capabilities из `.sdd.yaml.creates` и путей в `openspec/specs/`.
- В `sdd:apply` добавляется **обязательный блок** `## Как проверить` сразу после `## Реализованные фичи`. Для каждой реализованной фичи блок отвечает на три вопроса:
  - **Что** проверяем — capability или конкретный сценарий из `test-plan.md`.
  - **Где** проверяем — путь к файлу, URL, имя теста, директория артефакта.
  - **Как** проверяем — точная команда (`make test`, `pytest path::test`, `curl …`) или шаги ручной проверки + ожидаемый результат.
  Этот блок есть только в `sdd:apply` (для `contradiction` и `archive` он не имеет смысла — там нечего «проверять» в этом смысле).
- **Принцип отделения**: «Решено самостоятельно» снимает путаницу между «не нужно от пользователя» и «надо ответить» — автономные решения визуально и структурно отделены от блока вопросов соседним заголовком, а не помещены в прозу. «Прочее» — заметки без принятого решения; отличается от «Решено самостоятельно» отсутствием маркера `→`.
- **Правила коммуникационного стиля** (применяются скиллами при формировании финального отчёта и при общении с пользователем в рамках своего шага):
  1. **По умолчанию — действие, не вопрос.** Любую развилку, которую Claude может закрыть сам (имя, путь, формат, формулировка), он закрывает автономно и фиксирует в `## Решено самостоятельно`. Вопрос — только если ответ физически невозможно вывести без знания/преференции пользователя.
  2. **Форма ответа = форма запроса.** Семиблочный формат отчёта — **только** в финальном выводе `sdd:apply` / `sdd:contradiction` / `sdd:archive`. В обычной переписке (промежуточные шаги, уточнения, мета-вопросы) — проза, 2–3 предложения, без markdown-секций.
  3. **Действие первое, отчёт после.** Если задача требует правок (`Edit`/`Write`), скилл выполняет правки первыми и затем отчитывается 1–2 строками «что сделано», а не «я собираюсь сделать X». Исключение — действия с реальным побочным эффектом (push, drop, deploy), там — подтверждение.
  4. **Ноль внутреннего жаргона в user-facing полях.** В блоках «Описание» и «Решено самостоятельно» SHALL NOT появляться термины внутреннего инструментария (`hard issue`, `drift_score`, `residual_risk`, `SSOT`, `pointer-rewrite`, `Mandatory-блок N`) — кроме случаев, когда пользователь явно их запросил (например, `sdd:contradiction` показывает детализированный отчёт детекторов внутри `## Технические статусы`). Переводить на естественный язык пользователя.

## Impact

**`skills/sdd/apply.md`** — добавляется финальный шаг: 5 обязательных блоков (Технические статусы / Описание / Реализованные фичи / Как проверить / Вопросы к пользователю — последний) + 2 опциональных между ними (Решено самостоятельно / Прочее). Зеркало в `.claude/skills/sdd/apply.md` синхронизируется.

**`skills/sdd/contradiction.md`** — поверх существующего детализированного отчёта добавляется обязательный user-facing wrapper в том же порядке (Технические статусы (содержит детализированный отчёт verbatim) / Описание / Найденные противоречия / опц. Решено самостоятельно / опц. Прочее / Вопросы к пользователю). Зеркало в `.claude/skills/sdd/contradiction.md` синхронизируется.

**`skills/sdd/archive.md`** — добавляется финальный шаг с тем же порядком блоков (Технические статусы / Описание / Архивированные артефакты / опц. Решено самостоятельно / опц. Прочее / Вопросы к пользователю). Зеркало в `.claude/skills/sdd/archive.md` синхронизируется.

**Прочие скиллы, обращённые к пользователю** — формат вывода в текущей итерации регламентируется только для трёх перечисленных. Остальные скиллы — Non-Goal этого change'а (см. `design.md` → Goals/Non-Goals).
