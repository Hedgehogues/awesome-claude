## Context

См. `proposal.md` → ## Why (мотивация и текущее состояние вывода).

`sdd:apply` сейчас возвращает свободный текст: смесь технических фактов, прозы и вопросов. По завершении нет канонического списка «что именно реализовано». `.sdd.yaml.creates` — единственный авторитетный список capabilities, затронутых change'ом, и `tasks.md` — единственный источник состояния задач (done/open).

`sdd:contradiction` уже имеет жёстко регламентированный формат отчёта детекторов (Hard issues / Cascade predictions / CONVERGE pass / Soft warnings / Subjects covered / Summary). Существующий формат — внутренний, детализированный, ориентированный на инструмент. Поверх него отсутствует короткая user-facing выжимка с явным разделением «технические статусы / описание / вопросы / итог».

`sdd:archive` сейчас выводит то, что напечатает `openspec-archive-change` + сообщения о копировании test-plan'ов. Нет канонического списка «что заархивировано» в формате capability-уровня.

## Goals / Non-Goals

**Goals:**
- Единая, фиксированная, парсимая структура user-facing вывода для трёх скиллов `sdd:apply`, `sdd:contradiction`, `sdd:archive`: три общих блока + один per-skill блок «Итоги».
- Чёткое разделение «вопросов к пользователю» (blocking) и «прочих вопросов» (non-blocking).
- Список «итогов» (фичи / противоречия / архивированные артефакты) строится из структурированных источников (`.sdd.yaml`, `tasks.md`, отчёт детекторов), без LLM-реконструкции.
- Существующий детализированный отчёт `sdd:contradiction` сохраняется как есть и располагается внутри блока «Технические статусы» — не переписывается, не дублируется.

**Non-Goals:**
- Регламентация вывода всех остальных скиллов (`dev:*`, `report:*`, `research:*`, прочих `sdd:*`). В этой итерации — только три перечисленных.
- Локализация формата (шаблон фиксированно на русском, что соответствует языку проекта).
- Машиночитаемый вывод (JSON/YAML) — оставлен на будущее, текущий формат — markdown.
- Изменение содержания / семантики детекторов `sdd:contradiction` — меняется только обёртка вывода.

## Technical Approach

Каждый из трёх скиллов (`sdd:apply`, `sdd:contradiction`, `sdd:archive`) добавляет финальный шаг «User-facing отчёт». Шаг формирует markdown в **фиксированном порядке**: всё информационное и не требующее ответа — сверху, блок «Вопросы к пользователю» — последний. Между обязательными блоками вставляются (если применимо) опциональные.

**Канонический порядок блоков (top-to-bottom):**

1. `## Технические статусы` (mandatory) — буллеты, каждая строка = один факт. Допускается: путь к файлу, результат теста, изменение `index.yaml`, exit code команды, путь к заархивированному артефакту, имена применённых детекторов. Запрещается: проза, оценки, общие фразы. Для `sdd:contradiction` сюда вкладывается ранее существующий детализированный отчёт детекторов целиком, без изменений.
2. `## Описание` (mandatory) — 2–5 предложений прозы. Что произошло на уровне фич / противоречий / архивации. Без перечисления файлов.
3. `## <per-skill итог>` (mandatory) — per-skill (детали ниже). Может быть `_нет_`, заголовок всегда присутствует.
4. `## Как проверить` (mandatory **только для `sdd:apply`**, отсутствует у `contradiction` и `archive`) — для каждой реализованной capability/фичи структурированная инструкция «что / где / как» (детали в D7).
5. `## Решено самостоятельно` (optional) — формат `<вопрос-кратко> → <решение-кратко>`. Опускается полностью, если решений не было.
6. `## Прочее` (optional) — риторические замечания, follow-up идеи. Опускается полностью, если их нет.
7. `## Вопросы к пользователю` (mandatory, **always last**) — только то, что выходит за компетенцию Claude. **Последняя строка — всегда CTA** (см. D2).

**Per-skill содержимое блока 3 «Итоги»:**

- `sdd:apply` → `## Реализованные фичи` — нумерованный список. Источник: `.sdd.yaml.creates`. Для каждой capability — статус (`done` | `partial`) и одна строка из `proposal.md → ### New Capabilities`.
- `sdd:contradiction` → `## Найденные противоречия` — компактный сводный список из Summary детализированного отчёта: общее число hard issues по классам и общее число warnings. Если hard issues отсутствуют — `_нет_`.
- `sdd:archive` → `## Архивированные артефакты` — нумерованный список заархивированных capabilities из `.sdd.yaml.creates`. Для каждой — путь в `openspec/specs/<capability>/spec.md` и `openspec/specs/<capability>/test-plan.md`. Если `.sdd.yaml.creates` пустое — `_нет_`.

**Принцип отделения вопросов:** ничто, не требующее реакции пользователя, не должно стоять ниже блока 6. Это даёт пользователю одно правило взгляда: «прокрутка до конца — там вопросы и CTA».

## Architecture Decisions

### D1: Жёсткая структура с фиксированными заголовками и порядком

Выбран фиксированный набор и **фиксированный порядок** markdown-секций (см. Technical Approach). Заголовки 1, 2, 4, 5, 6 одинаковы для всех трёх скиллов; заголовок блока 3 — per-skill (`## Реализованные фичи` / `## Найденные противоречия` / `## Архивированные артефакты`). Mandatory-блоки: 1, 2, 3, 6. Optional: 4, 5.

**Альтернатива:** структурированный формат (JSON/YAML) или таблица. **Отклонено:** markdown совместим с текущим форматом ответов Claude Code; жёсткие заголовки обеспечивают парсимость без отдельного парсера.

### D2: Блок «Вопросы к пользователю» — последний, с обязательным CTA

Блок «Вопросы к пользователю» SHALL быть **последним** в отчёте. Любой контент, не требующий реакции пользователя, помещается выше него (см. D6). Цель — дать пользователю одно правило взгляда: «прокрутка до конца — там вопросы и CTA».

Содержимое блока: только вопросы, чей ответ требует знания/предпочтения, которыми Claude не обладает (бизнес-приоритеты, доменные решения, личные преференции, развилки с равновесными последствиями). Всё, что Claude может закрыть автономно, — в блок `## Решено самостоятельно` (D2a). Всё, что не требует решения, — в `## Прочее` или опускается.

**Call-to-action — обязателен.** Последняя строка блока — всегда CTA. По умолчанию: `Продолжаем по флоу? (да / нет)`. Если есть user-only вопросы, они нумеруются первыми, CTA — финальная строка с продолжающимся номером. Минимум блока — одна строка `1. Продолжаем по флоу? (да / нет)` (если других вопросов нет).

**Альтернатива A:** все вопросы вперемешку с информационным контентом по дате/важности. **Отклонено:** пользователь не должен сканировать отчёт в поисках «есть ли что-то, на что я должен ответить». Якорь в конце — однозначный сигнал.

**Альтернатива B:** разрешить пустой блок (без дефолтного CTA). **Отклонено:** в SDD-флоу следующий шаг (`apply` → `contradiction` → `archive`) не запускается автоматически — пользователь должен явно подтвердить переход. CTA делает это явным.

**Альтернатива C:** CTA — отдельный седьмой блок. **Отклонено:** CTA — это вопрос к пользователю; разделение размывает контракт «вопросы → ответы».

### D2a: Блок `## Решено самостоятельно`

Каждая строка блока — `<вопрос-кратко> → <принятое решение-кратко>`. Пример: `имя capability шире скоупа → переименование отложено в follow-up`. Назначение — отделить «решено мной» от «нужно от вас» соседними блоками, чтобы пользователь не сканировал прозу в поисках того, на что обязан реагировать.

**Альтернатива:** не вводить отдельный блок; помещать автономные решения в `## Описание`. **Отклонено:** в прозе они теряются и неотличимы от констатации фактов; пользователь начинает перечитывать на предмет «это вопрос ко мне или нет?».

### D3: Содержимое блока «Итоги» строится из структурированных источников

Per-skill источники истины:
- `sdd:apply` → `.sdd.yaml.creates` × `tasks.md`. Capability считается `done`, если все задачи в `tasks.md`, относящиеся к ней, отмечены `[x]`. Если есть открытые — `partial`.
- `sdd:contradiction` → Summary существующего детализированного отчёта детекторов (численные счётчики hard / warnings).
- `sdd:archive` → `.sdd.yaml.creates` + проверка наличия файлов в `openspec/specs/<capability>/`.

**Альтернатива:** реконструировать списки из истории сессии через LLM. **Отклонено:** недетерминированно, противоречит правилу «структурированные данные через скрипт» из `claude-way.md`.

### D4: Scope — три workflow-скилла; остальные — отдельный change

В этом change затронуты только `sdd:apply`, `sdd:contradiction`, `sdd:archive`. Остальные скиллы (`sdd:propose`, `sdd:explore`, `dev:*`, `report:*`, `research:*` и пр.) — Non-Goal.

**Альтернатива:** ввести правило сразу для всех скиллов. **Отклонено:** слишком большой blast radius. Три перечисленных скилла — финальные шаги SDD-цикла, у которых наиболее критично явное разделение «что сделано / что нужно от пользователя». Остальные расширяются отдельным change'ем после обкатки.

### D5: Существующий отчёт `sdd:contradiction` сохраняется как есть

Детализированный отчёт детекторов (Hard issues / Soft warnings / Subjects covered / Summary) — внутренний artefact инструмента; ломать его формат рискованно (он уже консьюмится `sdd:change-verify` и сценариями отладки). Поэтому он целиком вкладывается в блок `## Технические статусы` user-facing отчёта без изменений. Поверх — добавляются три общих блока и компактный `## Найденные противоречия`.

**Альтернатива:** заменить существующий формат на трёхчастный. **Отклонено:** ломает downstream-потребителей и теряет диагностическую плотность.

### D6: Порядок блоков — «информационное сверху, требующее ответа снизу»

Принцип сортировки блоков по убыванию «обязательности реакции пользователя»:
- Не требуют реакции (информационные / автономные / факультативные): все блоки до последнего.
- Требуют реакции: последний блок (`## Вопросы к пользователю`).

Это даёт **детерминированное расположение CTA** — всегда на последней позиции отчёта. Пользователь, читая сверху вниз, проходит «факты → проза → итог → как проверить (apply) → автономные решения → заметки → вопросы», и финальная строка — призыв к действию.

**Альтернатива A:** хронологический порядок (что сделано → что обнаружено → вопросы). **Отклонено:** не работает для скиллов с разной природой шагов (`contradiction` диагностирует, `archive` финализирует).

**Альтернатива B:** вопросы вверху, чтобы их видели первыми. **Отклонено:** при крупных отчётах (особенно `sdd:contradiction` с детализированным отчётом внутри `## Технические статусы`) пользователь видит вопросы, но при чтении деталей теряет контекст вопроса. Якорение CTA в конце естественно: «дочитал — увидел вопрос — ответил».

### D7: Блок `## Как проверить` — обязательный для `sdd:apply`

После `sdd:apply` пользователь часто не знает, как именно убедиться, что фича работает. «Я доверяю отчёту» — слабая позиция; «я запустил вот эту команду и увидел вот это» — сильная. Поэтому в `sdd:apply` вводится обязательный блок `## Как проверить` сразу после `## Реализованные фичи`.

Каждая запись блока соответствует одной capability/фиче из `## Реализованные фичи` и содержит три явных поля:

- **Что** — что именно проверяется (capability имя или конкретный сценарий из `test-plan.md`).
- **Где** — путь к артефакту: файл, тест-кейс, URL, команда, директория.
- **Как** — точная команда (`make test`, `pytest <path>::<test>`, `bash <script>`) либо короткая последовательность ручных шагов + ожидаемый наблюдаемый результат.

**Формат записи (фиксированный):**

```markdown
1. **<имя фичи / capability>**
   - **Что:** <тестируемое поведение или сценарий>
   - **Где:** <путь / URL / команда>
   - **Как:** `<command>` → ожидание: <observable result>
```

**Источник данных.** Поля «Что» и «Где» автогенерируются: `apply_report.py` читает `test-plan.md` (поля `acceptance_criteria` + `## Scenarios`) и `tasks.md` (затронутые тестовые файлы и команды), формирует skeleton. Поле «Как» Claude дополняет точной командой исходя из контекста change'а — это автономное решение, фиксируется в `## Решено самостоятельно`, если требуется явный выбор (например, между `make test` и прямым `pytest`).

**Пустой случай.** Если `.sdd.yaml.creates` пустое или `test-plan.md` отсутствует — блок рендерится с `_нет_` (заголовок остаётся). Это редкий случай (обычно apply выполняется только когда test-plan заполнен), но защищает контракт «заголовок всегда есть».

**Альтернатива A:** не вводить блок, оставлять «как проверить» в прозе `## Описание`. **Отклонено:** проза теряется в потоке; пользователь не находит конкретную команду без перечитывания всего отчёта.

**Альтернатива B:** «Как проверить» — отдельный скилл (`sdd:verify`), вызывается явно. **Отклонено:** избыточная фрагментация флоу. Пользователь после apply хочет ответ на «как теперь убедиться» сразу, а не отдельным шагом.

**Альтернатива C:** вводить блок и для `contradiction` / `archive`. **Отклонено:** для `contradiction` «как проверить» = «перезапустить детектор», тривиально и не несёт ценности; для `archive` фактически нечего проверять (артефакт перемещён). Польза только в `apply`.

## Data Flow

**`sdd:apply` (шаг N+1: финальный отчёт):**

```
[шаги 1–4 завершены]
   │
   ▼
apply_report.py: читает .sdd.yaml.creates, парсит tasks.md (чекбоксы, file_facts),
                 читает test-plan.md (acceptance_criteria + ## Scenarios)
                 → JSON {capabilities[], file_facts[], verify[]}
   │
   ▼
Рендер блоков (порядок строгий):
   ## Технические статусы            ← file_facts + index.yaml updates
   ## Описание                       ← проза 2–5 предложений
   ## Реализованные фичи             ← capabilities со статусом done|partial
   ## Как проверить                  ← по фиче: «Что / Где / Как» (skeleton из verify[],
                                       команда заполняется LLM, фиксируется при необходимости
                                       в `## Решено самостоятельно`)
   ## Решено самостоятельно (опц.)   ← <вопрос> → <решение>
   ## Прочее (опц.)                  ← заметки без действия
   ## Вопросы к пользователю         ← user-only вопросы + CTA финальной строкой
                                       (минимум: «1. Продолжаем по флоу? (да / нет)»)
```

**`sdd:contradiction` (после Phase 5: финальный отчёт):**

```
[детекторы отработали → детализированный отчёт сформирован]
   │
   ▼
contradiction_summary.py: парсит Summary существующего отчёта → JSON {hard_counts, warning_counts, residual_risk}
   │
   ▼
Рендер блоков (порядок строгий):
   ## Технические статусы            ← существующий детализированный отчёт целиком (без изменений)
   ## Описание                       ← проза о найденных проблемах
   ## Найденные противоречия         ← компактный счётчик: hard=N, warnings=M, residual_risk=...
   ## Решено самостоятельно (опц.)
   ## Прочее (опц.)
   ## Вопросы к пользователю         ← decision-gate вопросы + CTA
                                       (минимум: «1. Продолжаем по флоу? (да / нет)»)
```

**`sdd:archive` (после шага 5: финальный отчёт):**

```
[openspec-archive-change отработал, test-plan.md скопированы]
   │
   ▼
archive_report.py: читает .sdd.yaml.creates, проверяет существование openspec/specs/<cap>/spec.md и test-plan.md → JSON {archived[]}
   │
   ▼
Рендер блоков (порядок строгий):
   ## Технические статусы            ← вывод openspec-archive-change + список скопированных файлов
   ## Описание                       ← проза о завершённом архивировании
   ## Архивированные артефакты       ← capabilities + пути к spec.md / test-plan.md
   ## Решено самостоятельно (опц.)
   ## Прочее (опц.)
   ## Вопросы к пользователю         ← обычно только CTA: «1. Продолжаем по флоу? (да / нет)»
```

Чтение `.sdd.yaml`, парсинг `tasks.md` и Summary детектора — структурированный доступ; согласно `claude-way.md` (правило «Структурированные данные»), реализуется тремя Python-скриптами в `skills/sdd/scripts/` (lifetime: `persistent`): `apply_report.py`, `contradiction_summary.py`, `archive_report.py`. Все три имеют общий код парсинга `.sdd.yaml` — выносится в shared-модуль `_sdd_yaml.py` рядом.

## Risks / Trade-offs

- **[Жёсткий шаблон избыточен для тривиальных запусков]** → допустимо: блок «Итоги» = `_нет_`, опциональные блоки опускаются. Минимальный обязательный объём — 4 заголовка (Технические статусы / Описание / Итог / Вопросы к пользователю с CTA).
- **[Скрипты дублируют часть логики `sdd:change-verify`]** → не дублируют: статус capability считается из `tasks.md`, а не из spec'ов. Логика поверхностная (булева агрегация чекбоксов / счёт строк отчёта).
- **[LLM может не следовать формату]** → semantic-кейсы в `skills/skill/cases/sdd/{apply,contradiction,archive}.md` фиксируют факт наличия четырёх заголовков; нарушения ловятся `skill:test-skill`.
- **[Существующий отчёт `sdd:contradiction` плохо вкладывается в `## Технические статусы` из-за длины]** → допустимо: блок «Технические статусы» может содержать вложенный код-блок или `<details>`-секцию. Объём не ограничивается; правило только про порядок и заголовки.
- **[Имя capability `structured-apply-output` теперь шире чем «apply»]** → известный риск; рассматривается переименование в follow-up change (см. Open Questions ниже).
