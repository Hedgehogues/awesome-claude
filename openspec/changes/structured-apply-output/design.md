## Context

См. `proposal.md` → ## Why (мотивация и текущее состояние вывода).

`sdd:apply` сейчас возвращает свободный текст: смесь технических фактов, прозы и вопросов. По завершении нет канонического списка «что именно реализовано». `.sdd.yaml.creates` — единственный авторитетный список capabilities, затронутых change'ом, и `tasks.md` — единственный источник состояния задач (done/open).

## Goals / Non-Goals

**Goals:**
- Фиксированный, парсимый формат финального вывода `sdd:apply`: три блока + список фич.
- Чёткое разделение «вопросов к пользователю» (blocking) и «прочих вопросов» (non-blocking).
- Список реализованных фич строится из `.sdd.yaml.creates` × `tasks.md` — без LLM-реконструкции.

**Non-Goals:**
- Регламентация вывода всех остальных скиллов (`dev:*`, `report:*`, `research:*`, прочих `sdd:*`). В этой итерации — только `sdd:apply`.
- Локализация формата (шаблон фиксированно на русском, что соответствует языку проекта).
- Машиночитаемый вывод (JSON/YAML) — оставлен на будущее, текущий формат — markdown.

## Technical Approach

После выполнения шага 4 (обновление `index.yaml`) в `sdd:apply` добавляется обязательный шаг 5 «Финальный отчёт». Шаг 5 формирует ровно четыре markdown-блока, в фиксированном порядке, с фиксированными заголовками:

1. `## Технические статусы` — буллеты, каждая строка = один факт. Допускается: путь к файлу, результат теста, изменение `index.yaml`, exit code команды. Запрещается: проза, оценки, общие фразы.
2. `## Описание` — 2–5 предложений прозы. Что произошло на уровне фич. Без перечисления файлов.
3. `## Вопросы к пользователю` — нумерованный список. Каждый вопрос проходит критерий блокировки: «без ответа на этот вопрос пользователь не может продолжить». Если таких вопросов нет — секция содержит строку `_нет_`.
4. `## Реализованные фичи` — нумерованный список. Источник: `.sdd.yaml.creates`. Для каждой capability — статус (`done` | `partial`) и одна строка из `proposal.md → ### New Capabilities`.

«Прочие вопросы» (риторические, размышления, follow-up идеи) выносятся в отдельный блок `## Прочее` либо не выводятся пользователю вовсе. Этот блок не входит в обязательную тройку и не блокирует пользователя.

## Architecture Decisions

### D1: Жёсткая структура с фиксированными заголовками

Выбрана последовательность из четырёх markdown-секций с фиксированными русскоязычными заголовками: `## Технические статусы`, `## Описание`, `## Вопросы к пользователю`, `## Реализованные фичи`.

**Альтернатива:** структурированный формат (JSON/YAML) или таблица. **Отклонено:** markdown совместим с текущим форматом ответов Claude Code; жёсткие заголовки обеспечивают парсимость без отдельного парсера.

### D2: Критерий «вопрос обращён к пользователю»

Вопрос попадает в блок 3 только если соответствует строгому критерию: **«без ответа пользователь не может продолжить»**. Всё остальное (риторические, follow-up, exploratory) — в блок «Прочее» или скрывается.

**Альтернатива:** все вопросы в одном блоке с маркерами приоритета. **Отклонено:** пользователь должен сразу видеть, что от него требуется, без сортировки маркеров. Минимизация числа вопросов в блоке 3 — самостоятельная цель.

### D3: Список фич строится из `.sdd.yaml`, не из прозы

Источник истины — `.sdd.yaml.creates` (capabilities) × `tasks.md` (статус задач). Capability считается `done`, если все задачи в `tasks.md`, относящиеся к ней, отмечены `[x]`. Если есть открытые — `partial`.

**Альтернатива:** реконструировать список из истории сессии через LLM. **Отклонено:** недетерминированно, противоречит правилу «структурированные данные через скрипт» из `claude-way.md`.

### D4: Scope только `sdd:apply`

Ввод трёхчастного формата для остальных скиллов — отдельный change. В этом — только `sdd:apply` и связанная capability `structured-apply-output`.

**Альтернатива:** ввести правило сразу для всех скиллов. **Отклонено:** слишком большой blast radius; разумнее обкатать формат на одном скилле, затем расширить.

## Data Flow

```
sdd:apply (шаг 4 завершён)
   │
   ▼
Прочитать .sdd.yaml.creates → [capability_a, capability_b, ...]
   │
   ▼
Для каждой capability: пройтись по tasks.md, определить done|partial
   │
   ▼
Собрать факты: список изменённых файлов (из шагов 1–4), результаты тестов (если запускались), записи в index.yaml
   │
   ▼
Сформировать вывод в порядке:
   ## Технические статусы  ← факты
   ## Описание             ← проза 2–5 предложений
   ## Вопросы к пользователю ← блокирующие вопросы или «_нет_»
   ## Реализованные фичи   ← capability + статус + одна строка
   ## Прочее (опционально)  ← риторические/follow-up
```

Чтение `.sdd.yaml` и парсинг `tasks.md` — структурированный доступ; согласно `claude-way.md` (правило «Структурированные данные»), реализуется отдельным Python-скриптом `skills/sdd/scripts/apply_report.py` (lifetime: `persistent`).

## Risks / Trade-offs

- **[Жёсткий шаблон может быть избыточен для тривиальных apply]** → допустимо: блок 3 = `_нет_`, блок «Прочее» опускается. Минимальный обязательный объём — 4 заголовка.
- **[`apply_report.py` дублирует часть логики `sdd:archive`/`sdd:change-verify`]** → не дублирует: статус capability считается из `tasks.md`, а не из spec'ов. Логика поверхностная (булева агрегация чекбоксов).
- **[LLM может не следовать формату]** → semantic-кейс в `skills/skill/cases/sdd/apply.md` фиксирует факт наличия четырёх заголовков; нарушения ловятся `skill:test-skill`.
