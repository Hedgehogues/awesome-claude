## Context

см. `proposal.md` → Why (мотивация и описание проблемы Agent tool)

## Goals / Non-Goals

**Goals:**
- Гарантировать сохранность форматированного пользовательского вывода скиллов при любом способе запуска (прямой / через Agent tool)
- Добавить write-then-replay как стандартный паттерн с правилом в `claude-way.md`
- Сохранить разделение: технический лог (`-<ts>.md`) и пользовательский вывод (`-<ts>-output.md`) — два разных файла

**Non-Goals:**
- Изменять формат или содержимое форматированного вывода
- Применять паттерн к `sdd:propose` (его финальный вывод — prose, не многосекционный блок)
- Изменять существующие детали технического лога

## Decisions

**D1: Write-then-replay как последняя фаза скилла**
Форматированные блоки записываются в `.log/<skill>-<ts>-output.md` через bash-heredoc, затем скилл выводит `cat` этого файла. Это атомарно с точки зрения скилла: write происходит до `cat`, файл всегда существует к моменту replay.

*Альтернатива:* записывать только в конце — отклонено, т.к. при падении на `cat` вывод всё равно теряется. Heredoc пишет сначала, потом воспроизводит.

**D2: Два отдельных файла в `.log/`**
см. `specs/write-then-replay-output/spec.md` → Requirement: Технический лог и пользовательский вывод — отдельные файлы.

*Альтернатива:* один файл с двумя секциями — отклонено, это смешивает аудитории.

**D3: Применить к трём скиллам: apply, contradiction, archive**
У каждого из них есть жёстко структурированный финальный блок (`## Описание`, `## Реализованные фичи` и т.д.). У `sdd:propose` итоговый вывод — проза + ссылка на лог, паттерн не нужен.

**D4: Правило write-then-replay фиксируется в `claude-way.md`**
`claude-way.md` — единственное место для принципов поведения Claude Code-скиллов в awesome-claude (там уже живут «Единственный интерфейс», «Жизненный цикл артефакта», «Структурированные данные»). Write-then-replay как архитектурный принцип скилла туда же. Альтернатива: добавить в CLAUDE.md или отдельный rules-файл — отклонено, дробит связанные принципы.

## Risks / Trade-offs

- [Риск] Файл `-output.md` остаётся на диске после завершения → Не проблема: `.log/` исключён из git через `**/.log/`
- [Риск] При падении скилла до write-фазы файл не создаётся → Приемлемо: это означает ошибку до вывода, пользователь увидит stack trace или ничего — не хуже текущего поведения
- [Trade-off] Дублирование: форматированный вывод идёт и в диалог, и в файл → осознанное решение ради надёжности при Agent tool запуске
