## Context

Четыре SDD-скилла (`sdd:propose`, `sdd:contradiction`, `sdd:apply`, `sdd:archive`) имеют схожую структуру финального вывода: технический блок (`## Технические статусы`) + пользовательские секции. Технический блок — verbatim dump детекторов, exit-кодов и JSON — нужен для аудита и отладки, но не для принятия решений. Пользовательские секции написаны в терминах детекторов и путей к файлам, которые пользователь не знает.

Аналог в других системах: CI-лог (сохраняется в артефактах, не в чате) + краткий статус в Pull Request (виден в интерфейсе).

## Goals / Non-Goals

**Goals:**
- Переместить технический вывод каждого скилла в timestamped лог-файл внутри change-директории
- Каждый запуск создаёт отдельный файл; история запусков сохраняется
- `.log/` не коммитится — это runtime-артефакт, не часть change'а
- В `contradiction`-логе явно отображать drift-веса с комментариями («1 hard issue = 10 баллов, 4 warning = 4 балла, итого 14 — high drift»)
- Пользовательские секции `## Что делать` и `## Вопросы к пользователю` — только задача и решение, без файловых путей и жаргона детекторов
- Ссылка на лог — одна строка в конце диалогового вывода

**Non-Goals:**
- Агрегация или поиск по логам (лог — плоский файл, не БД)
- Ротация/очистка логов (достаточно `.gitignore`)
- Изменение логики детекторов, верификации или state-машины

## Decisions

**D1: Формат имени лог-файла — `<skill>-<ISO8601-compact>.md`**
Timestamp в имени файла даёт сортировку по дате без чтения содержимого. ISO8601 без разделителей (`20260503T142301`) — короткий, валидный для файловых систем. Альтернатива (sequential числа) требует чтения директории; UUID — нечитаем.

**D2: Запись лога — один Bash tool call с heredoc**
Каждый скилл инструктирует Claude записать лог через Bash tool после формирования отчёта. Никакого нового Python-скрипта не нужно: `mkdir -p` + `cat > file` достаточно для плоского Markdown-файла.

**D3: Структура лога фиксирована для каждого скилла**
`contradiction`-лог: frontmatter (change, timestamp, exit_semantics), drift-таблица с весами, полный вывод детекторов.
`apply`-лог: frontmatter, L1/L2/L3 результаты по задачам, verify-вердикт.
`propose`-лог: frontmatter, шаги выполнения, найденные capability-пересечения.
`archive`-лог: frontmatter, список архивируемых файлов, результаты проверок.

**D4: `## Что делать` генерируется по правилу "один warning → одна строка действия"**
Для каждого warning скилл формирует строку вида: `- [ ] <действие на русском без технических терминов>`. Правило: описывать ЧТО изменить в тексте change'а, не КАК файл называется. Например: «Убери дублирующийся мотивационный абзац из раздела Context» вместо «design.md:3 redundancy pointer-rewrite».

**D5: `.log/` в `.gitignore` — паттерн `**/.log/`**
Паттерн покрывает все change-директории. Альтернатива (per-change `.gitignore`) — overhead без пользы.

## Risks / Trade-offs

[Risk] Лог-файл может быть большим (contradiction с 30 capabilities — ~200 строк) → Mitigation: Markdown-файл, всё равно не коммитится; размер не критичен.

[Risk] Bash tool call для записи файла может не выполниться если change-директория не существует → Mitigation: `mkdir -p` в команде записи; это стандартная практика.

[Risk] Зеркальные копии скиллов в `skills/sdd/*/skill.md` (корень репо) придётся обновлять синхронно → Mitigation: задачи в tasks.md явно включают зеркала; синхронность — ответственность автора change'а.
