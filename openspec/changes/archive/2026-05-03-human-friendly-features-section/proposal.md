## Why

Две секции в выводе `sdd:apply` — `## Реализованные фичи` и `## Как проверить` — содержат технический жаргон, который непонятен без знания внутренней структуры change'а.

**Реализованные фичи** показывает:

```
- sdd-state-skill-hooks — partial (integration тесты требуют live-run; bump-version отложен)
```

- `sdd-state-skill-hooks` — kebab-case идентификатор из `.sdd.yaml`, не описывающий что сделано
- `partial` — технический термин без перевода
- Причина — сырые тексты незакрытых задач из `tasks.md` с CLI-командами и жаргоном

**Как проверить** показывает:

```
1. sdd-state-skill-hooks
   - Что: skill.md файлы не содержат прямых transition-вызовов
   - Как: grep -r "state\.py transition" skills/sdd/*/skill.md → ожидание: 0 совпадений
2. sdd-state-skill-hooks
   - Как: cat .claude/settings.json → поле PostToolUse[0].matcher = "Skill"
```

- Capability снова техническим именем, хотя у него есть человекочитаемый title
- «Что» описывает реализацию, а не что нужно проверить с точки зрения пользователя
- «Как» — сырые shell-команды с регулярными выражениями и JSON-путями

Пользователь должен видеть человекочитаемый статус работы и понятные инструкции по проверке.

См. `.sdd.yaml` для capability-декларации.

## What Changes

- Добавить в `.sdd.yaml` необязательное поле `title` для каждого capability в `creates` (обратно совместимо: запись остаётся строкой ИЛИ становится объектом `{name: str, title: str}`)
- Обновить `apply_report.py`: если capability записан как объект — включать `title` в JSON; для partial-capabilities возвращать `incomplete_count` (число незакрытых задач) вместо сырых текстов
- Обновить шаг рендера «Реализованные фичи» в `sdd:apply`: отображать `title` (если есть) вместо `name`; статусы переводить: `done` → «готово», `partial` → «частично (N задач не завершено)»
- Обновить шаг рендера «Как проверить» в `sdd:apply`: заголовок capability использует `title`; поле «Что» описывает ожидаемое поведение с точки зрения пользователя, а не реализацию; поле «Как» начинается с понятного ожидания, команда — опциональное уточнение
- Обновить `sdd:contradiction`: каждая строка в `## Что делать` заканчивается подсказкой `(непонятно — скажи «поясни», слишком очевидно — скажи «детали»)` — пользователь может запросить упрощение или детализацию любого пункта
- Обновить `sdd:propose`: если новый capability добавлен в `creates` как строка — предложить добавить `title`

## Capabilities

### New Capabilities

- `human-friendly-features-section`: секции `## Реализованные фичи` и `## Как проверить` в `sdd:apply` используют человекочитаемые заголовки, статусы и описания проверок вместо технических идентификаторов и сырых команд

## Impact

**`skills/sdd/scripts/apply_report.py`** — читать `title` из `.sdd.yaml` если capability записан как объект; возвращать `{name, title?, status, incomplete_count}` для каждого capability.

**`skills/sdd/apply/skill.md`** — изменить шаг рендера «Реализованные фичи»: использовать `title` (или `name` если `title` отсутствует); статусы на русском; для partial — «(N задач не завершено)».

**`.claude/skills/sdd/apply/skill.md`** — зеркало.

**`skills/sdd/apply/skill.md`** (шаг «Как проверить») — заголовок capability использует `title`; поле «Что» = ожидаемое поведение; поле «Как» = понятное описание + необязательная команда.

**`skills/sdd/propose/skill.md`** — добавить подсказку: после создания `creates`-записей проверить наличие `title` и предложить его заполнить если запись осталась строкой.

**`.claude/skills/sdd/propose/skill.md`** — зеркало.
