## Context

`sdd:apply` завершается блоком `## Реализованные фичи`, который строится из `apply_report.py`. Скрипт читает `creates: list[str]` из `.sdd.yaml` и сопоставляет их с чекбоксами `tasks.md`, возвращая `{name: str, status: "done"|"partial"}`. Скилл рендерит это дословно, подставляя к partial-записям сырые тексты незавершённых задач.

Результат нечитаем: пользователь видит kebab-case идентификаторы и технические детали задач там, где должен видеть статус своей работы на естественном языке.

Смежные изменения: `structured-apply-output` зафиксировал структуру блока; `human-friendly-skill-output` сделал человекочитаемыми `## Что делать` и `## Вопросы к пользователю`. Секция `## Реализованные фичи` осталась за рамками обоих.

## Goals / Non-Goals

**Goals:**
- Каждый capability в `## Реализованные фичи` отображается с человекочитаемым заголовком
- Статусы `done`/`partial` рендерятся на русском языке
- Для partial-capability причина формулируется в терминах «N задач не завершено», а не сырого списка задач
- Расширение `.sdd.yaml` обратно совместимо: существующие `creates: [string]` продолжают работать

**Non-Goals:**
- Автоматическая генерация `title` из `name` (kebab-to-human — слишком ненадёжно для технических имён)
- Изменение формата `creates` для `merges-into` — в этой итерации только `creates`
- Локализация статусов в других языках
- Изменение схемы `.sdd-state.yaml` или `tasks.md`

## Decisions

**D1: Расширение схемы `.sdd.yaml` через union-тип, не через новое поле верхнего уровня.**

`creates` остаётся списком, но каждый элемент может быть либо строкой (как сейчас), либо объектом `{name: str, title: str}`. Причина: минимальный diff для существующих change'ов; парсер `_sdd_yaml.py` нормализует оба варианта в `{name, title?}` для единообразия.

**D2: `incomplete_count` вместо текстов задач.**

`apply_report.py` для partial-capability считает незакрытые чекбоксы группы и возвращает число. Скилл рендерит «(N задач не завершено)». Причина: тексты задач — внутренняя реализация; пользователю важна степень готовности, а не список «что именно не сделано» (это есть в `tasks.md`).

**D3: `title` — обязательная рекомендация, не ошибка.**

`sdd:propose` предлагает добавить `title` если capability остался строкой, но не блокирует выполнение. Причина: `title` — улучшение UX, не инвариант системы; принуждение сломает текущий поток.

**D4: «Как проверить» описывает ожидаемое поведение, не реализацию.**

Поле «Что» в каждом пункте `## Как проверить` формулируется с точки зрения пользователя («что я увижу»), а не разработчика («что есть в коде»). Поле «Как» начинается с предложения-ожидания на русском языке; shell-команда для проверки идёт после, как необязательное уточнение. Причина: `## Как проверить` — инструкция для владельца change'а на стадии `verifying`; он не обязан знать внутреннюю структуру скриптов.

**D5: Технические развилки Claude закрывает автономно, в вопросы пользователю не выносит.**

Если в ходе работы скилла возникает развилка между несколькими способами реализации, разница которых видна только в коде, Claude выбирает сам и фиксирует выбор в `## Решено самостоятельно`. В `## Вопросы к пользователю` попадают только вопросы, на которые можно ответить без знания кодовой базы. Причина: пользователь не должен изучать код, чтобы ответить на технический вопрос; технические выборы — зона ответственности Claude.

**D6: CTA-подсказка в «Что делать» и «Вопросы к пользователю» — единственный способ взаимодействия с выводом скилла.**

Каждый пункт `## Что делать` и каждый вопрос `## Вопросы к пользователю` заканчиваются фиксированной подсказкой `(непонятно — скажи «поясни», слишком очевидно — скажи «детали»)`. Причина: без явного call-to-action пользователь не знает, что может попросить сократить или детализировать; подсказка делает взаимодействие явным и симметричным — работает в обе стороны (упрощение и детализация).

## Risks / Trade-offs

**R1: Расхождение `title` и реального содержания capability.**
Если автор напишет misleading title — секция станет ещё менее понятной. Митигация: `sdd:propose` предлагает title как подсказку, финальное решение за автором.

**R2: Backward compatibility — сценарий чтения.**
Существующий код, читающий `creates` как `list[str]`, сломается на объектах. Митигация: все читатели идут через нормализатор `_sdd_yaml.py.read_creates()` → всегда возвращает `list[str]`; новая функция `read_creates_with_meta()` → возвращает `list[{name, title?}]`. Только `apply_report.py` использует новую функцию.
