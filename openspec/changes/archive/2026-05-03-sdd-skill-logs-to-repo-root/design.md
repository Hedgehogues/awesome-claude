## Context

Логирование skill-runs сейчас реализовано только в `sdd:contradiction` с путём `openspec/changes/<name>/.log/<skill>-<TS>.md`. Три остальных SDD-скилла (apply, propose, archive) шаг логирования описывают в skill.md, но либо не реализуют, либо реализуют непоследовательно.

Текущая схема хранения `.log/` внутри change-директории означает, что для просмотра всех runs нужно обходить каждую директорию в `openspec/changes/`. При большом количестве changes диагностика становится неудобной.

## Goals / Non-Goals

**Goals:**
- Единый `.logs/` в корне репозитория — одна точка входа для всех skill-run логов
- Структура `.logs/<change-name>/<skill>-<TS>.md` сохраняет привязку к change
- Все четыре SDD-скилла пишут логи по единому паттерну
- `.logs/` исключён из git через `.gitignore`

**Non-Goals:**
- Не меняем формат содержимого лог-файлов (только путь)
- Не добавляем ротацию или retention policy
- Не трогаем скиллы вне sdd-namespace

## Decisions

### Решение 1: `.logs/` в корне репозитория, не в `openspec/`

Альтернатива — `openspec/.logs/` внутри openspec-дерева. Отклонено: `.logs/` в корне явно сигнализирует о runtime-артефакте, не о части спеки. Стандартная конвенция для ephemeral runtime данных — в корне проекта рядом с `.git/`.

### Решение 2: сохранить поддиректорию `<change-name>/` внутри `.logs/`

Альтернатива — плоская структура `<skill>-<TS>.md` без поддиректорий. Отклонено: при множестве changes и skills файлы смешаются, имя скилла не даёт контекста без имени change.

### Решение 3: добавить лог-шаг в apply и archive как финальный шаг

apply и archive — long-running операции с множеством шагов. Лог пишется в конце как итоговый отчёт о run. Это согласуется с тем, как contradiction пишет лог после анализа.

### Решение 4: обновить `.gitignore` — заменить `**/.log/` на `.logs/`

Перенос с per-change `.log/` на репозиторный `.logs/` требует синхронного обновления `.gitignore`. Паттерн `**/.log/` больше не нужен; `.logs/` покрывает всё одной строкой.

### Решение 5: обновить спеку `skill-run-log-archiving` как часть этого change

Спека описывает текущее поведение с путём `openspec/changes/<name>/.log/`. После реализации она должна отражать новый путь `.logs/<change-name>/` и расширенное покрытие (все 4 скилла). Обновление спеки — часть scope этого change, не отдельный follow-up.

## Risks / Trade-offs

- [Риск] Существующие логи в `openspec/changes/*/.log/` не мигрируются автоматически — Митигация: они ephemeral, gitignored; старые logs можно удалить вручную или проигнорировать
- [Риск] `.logs/` в корне добавляет шум в ls — Митигация: имя с точкой (hidden directory) минимизирует видимость
- [Trade-off] Единый `.logs/` создаёт зависимость от рабочей директории — скилл должен запускаться из корня репозитория. Это уже является предположением всех SDD-скиллов.
