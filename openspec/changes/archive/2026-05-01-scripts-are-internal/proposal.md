## Why

Awesome-claude — про Claude way: пользователь работает через Claude Code, а не через терминал. Это означает две вещи: (1) никаких прямых системных вызовов (bash, curl и т.д.) — любая операция, включая установку, выполняется через Claude; (2) никакой ручной работы в репозитории — файлы не редактируются напрямую, всё делается через Claude Code.

## What Changes

- Убрать из README все прямые системные вызовы (bash, curl и т.д.) — заменить на инструкцию через Claude Code
- Добавить в README явный раздел: единственный интерфейс — Claude Code; любые прямые системные вызовы — внутренняя реализация, не публичный API
- Добавить правило `rules/claude-way.md`: единственный интерфейс — Claude Code; устанавливается вместе с остальными правилами и автоматически загружается в контекст
- Перенести скрипты из `scripts/` к скиллам: каждый скрипт живёт рядом со скиллом, который его использует (`skills/<ns>/scripts/<script>.sh`)
- Зафиксировать в `rules/claude-way.md` принцип гранулярности скилла: последовательность шагов — это и есть bounded context; шагов в ней должно быть мало (1–2, максимум 3); скилл организует один шаг; артефакт живёт в рамках своего шага — следующий шаг принимает выходы одного или нескольких предыдущих шагов как свои входные артефакты; каноническая форма — 2 этапа по аналогии с ETL: сырой артефакт → преобразованный артефакт
- Зафиксировать в `rules/claude-way.md` правило жизненного цикла артефакта: каждый создаваемый артефакт должен иметь явно определённые границы (что входит, что нет) и время жизни
- Зафиксировать в `rules/claude-way.md` правило работы со структурированными данными: любой доступ (чтение и запись) к данным со схемой или повторяющейся структурой (YAML, JSON, TOML, INI, CSV, БД, `.versions`, `.manifest`) выполняется через Python-скрипт, не нативно

## Capabilities

### New Capabilities

- `user-interface-principle`: зафиксированный принцип Claude-way — пользователь взаимодействует только через Claude Code: никаких прямых системных вызовов (bash, curl и т.д.) и никакой ручной работы в репо напрямую
- `skill-step-principle`: последовательность шагов — это bounded context; скилл организует 1 шаг; шагов мало (1–2, не более 3); артефакт существует внутри шага и не переносится в следующий — следующий шаг получает на вход выходы одного или нескольких предыдущих шагов как свои собственные артефакты; каноническая форма — 2 этапа по аналогии с ETL: сырой артефакт → преобразованный артефакт
- `artifact-lifecycle-principle`: каждый создаваемый артефакт имеет явно декларированные границы области ответственности и время жизни (ephemeral / session / persistent)
- `structured-data-principle`: любой доступ к данным со схемой или повторяющейся структурой выполняется через Python-скрипт; скрипт — явный артефакт с декларированным lifetime

> Все четыре capability специфицированы совместно в `specs/user-interface-principle/spec.md` — они являются секциями одного правила `rules/claude-way.md` и семантически неделимы.
