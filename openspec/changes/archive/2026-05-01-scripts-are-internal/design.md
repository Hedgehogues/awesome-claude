## Context

Репозиторий awesome-claude предоставляет набор скиллов, устанавливаемых через `install.sh`. После установки для каждого namespace существует скилл `<ns>:bump-version`, который обновляет namespace до актуальной версии. Однако README.md содержит прямые вызовы `bash .claude/scripts/bump-namespace.sh <ns>` — пользователь видит внутреннюю реализацию там, где должен видеть только скилл.

## Goals / Non-Goals

**Goals:**
- README показывает только скиллы как способ обновления; bash-скрипты скрыты
- README содержит явный раздел: единственный интерфейс — Claude Code
- `rules/claude-way.md` фиксирует принцип и устанавливается в `.claude/rules/` вместе с остальными правилами
- Установка awesome-claude выполняется через Claude Code — пользователь говорит Claude установить репо, Claude клонирует и копирует файлы

**Non-Goals:**
- Не менять функциональное поведение скиллов и скриптов — только расположение
- Не касаться CI/CD — CI не использует эти скрипты

## Decisions

### D1 Установка через Claude Code — без curl

Claude Code уже установлен у пользователя до начала работы с awesome-claude. Поэтому установка выполняется через Claude: пользователь открывает Claude Code и просит установить awesome-claude с GitHub. Claude клонирует репо и копирует файлы. Прямой curl-вызов не нужен и не документируется.

### D2 Принцип фиксируется правилом, а не CLAUDE.md

Новый файл `rules/claude-way.md` описывает принцип Claude-way. При установке он копируется в `.claude/rules/` и автоматически загружается Claude Code в каждой сессии — без необходимости писать в CLAUDE.md пользователя и без конфликтов с его существующим CLAUDE.md.

### D3 README содержит явный раздел про интерфейс

Принцип Claude-way фиксируется в README явно, а не только подразумевается заменой команд. Пользователь читает раздел и понимает: скиллы — это намеренный выбор, не просто синтаксический сахар над скриптами.

### D4 Скрипты переносятся к скиллам, которые их используют

Все скрипты из `scripts/` кроме `install.sh` переносятся к скиллу-владельцу (`skills/<ns>/scripts/`). Перед переносом — аудит: для каждого скрипта определяется владелец; если неоднозначно — задаётся вопрос. Скилл ссылается на `${CLAUDE_SKILL_DIR}/scripts/` — путь не зависит от места установки.

### D5 Последовательность шагов — это bounded context

Последовательность скилл-вызовов является bounded context'ом: она определяет его границу. Скилл организует один шаг внутри этой последовательности. Количество шагов должно быть малым: 1–2, не более 3. Артефакт существует внутри своего шага и не переходит в следующий как общий объект — следующий шаг принимает выходы одного или нескольких предыдущих шагов как свои собственные входные артефакты. Каноническая форма — два этапа по аналогии с ETL: первый шаг производит сырой артефакт, второй преобразует его в итоговый. Длинная последовательность или разделяемые артефакты между шагами — сигнал к пересмотру границ.

### D6 Структурированные данные — только через Python-скрипт

Любой доступ (чтение и запись) к данным со схемой или повторяющейся структурой выполняется через Python-скрипт, не нативно. Граница: YAML, JSON, TOML, INI, CSV, базы данных, `.versions`, `.manifest`. Markdown, исходный код и free-text конфиги без схемы — не входят. Мотивация: данные могут быть распределены по нескольким файлам; скрипт делает полный обход, агрегацию и трансформацию явно, и его можно проверить до запуска. Lifetime скрипта соответствует lifetime трансформации: одноразовая операция — ephemeral, регулярная — persistent.

### D7 Каждый артефакт имеет явные границы и время жизни

Любой артефакт, создаваемый или изменяемый скиллом, обязан декларировать: (1) область ответственности — что он описывает и что намеренно не описывает; (2) время жизни — ephemeral (существует только в рамках одного вызова), session (существует на протяжении сессии Claude Code), persistent (записывается на диск и переживает сессию). Артефакт без явных границ нарушает принцип bounded context'а.

## Risks / Trade-offs

[Неполная миграция путей] → если хотя бы один из 4 скиллов (`skills/<ns>/bump-version.md`) или README не обновлён перед удалением `scripts/bump-namespace.sh`, вызов сломается. Митигация: перед удалением явно проверить все 9 вхождений (4 скилла + 4 строки README + 1 строка в listing структуры).

[`rules/claude-way.md` ещё не существует] → создаётся новый файл; содержимое должно быть лаконичным, чтобы не дублировать README.

[Неявная схема] → правило "структурированные данные через Python-скрипт" работает надёжно только если схема данных явно задекларирована (pydantic, JSON Schema, SQLAlchemy и т.п.). Если схема нигде не описана, Claude напишет скрипт с неверными предположениями о структуре — скрипт будет выглядеть корректно, но делать не то. Митигация: для этого репозитория структуры простые и предсказуемые (`.versions`, `.manifest`, файловое дерево); риск низкий. В будущем при усложнении схем — добавить явные pydantic-модели.
