## Context

SDD-скиллы (`contradiction`, `apply`, `archive`) принимают имя ченджа через `$ARGUMENTS`. При вызове без аргументов поведение не определено: скилл либо падает, либо задаёт неструктурированный вопрос. Пользователь вынужден явно указывать имя при каждом вызове.

Список ченджей уже присутствует в `openspec/changes/` — это единственный источник данных, который нужен для показа выбора пользователю.

Дополнительно: `sdd:contradiction` не имеет явного ограничения на область правок. Скилл может изменять рулы, скиллы и другие файлы — нарушая принцип bounded context ченджа.

## Goals / Non-Goals

**Goals:**
- При отсутствии `$ARGUMENTS` — показать список всех активных ченджей из `openspec/changes/` через `AskUserQuestion` и дождаться выбора пользователя.
- В `sdd:contradiction` — явное ограничение: все правки только внутри `openspec/changes/<name>/`. MUST NOT трогать любые файлы вне этой директории (подробности в D3).

**Non-Goals:**
- Не меняет поведение при передаче `$ARGUMENTS` — аргумент всегда имеет приоритет.
- Не вводит новых Python-скриптов — логика реализуется через инструкции в skill.md и Bash.
- Не распространяет scope-ограничение на `sdd:apply` и `sdd:archive` (они по природе правят только change-директорию).

## Decisions

**D1: Два режима определения ченджа.**

Шаг `Determine Change` работает в одном из двух режимов — в зависимости от уверенности в контексте:

**Режим A — контекст очевиден.**
Применяется когда: скилл вызван в продолжение того же разговора, в котором уже обсуждался конкретный чендж (например, только что завершился sdd:contradiction или пользователь явно упоминал имя ченджа). В этом режиме Claude **не вызывает никаких инструментов и не сканирует файлы**. Прямо из разговора показывает список вариантов:
```
1. Продолжить с `sdd-implicit-change-selection`
2. Выбрать другой чендж
```
Пользователь выбирает. Если «2» — переходит в режим B.

**Режим B — контекст неоднозначен.**
Применяется когда: скилл вызван в новой сессии или чендж из разговора не определить. Claude читает `openspec/changes/` и показывает список всех активных (не архивированных) ченджей через `AskUserQuestion`. Пользователь сам знает, что ему нужно — ранжирование не нужно.

**D2: Формат списка в режиме B.**
```
Выберите change для работы:
1. sdd-implicit-change-selection
2. sdd-state-manager
3. ...
```
Плоский нумерованный список, отсортированный алфавитно. Архивированные ченджи (`.sdd-state.yaml` со `stage: archived`) не показываются.

**D3: Scope-ограничение в sdd:contradiction.**
В skill.md добавляется явная инструкция: «Все правки выполняются исключительно внутри `openspec/changes/<name>/`. MUST NOT изменять файлы вне этой директории, включая `skills/`, `.claude/`, `openspec/specs/`.» Поведенческое ограничение — явная и обязательная инструкция в протоколе скилла.

**D4: Реализация через Bash + инструкции в skill.md, не через скрипт.**
Чтение списка директорий `openspec/changes/` — одна bash-команда. Новый Python-скрипт избыточен для этой задачи.

## Risks / Trade-offs

**R1: Длинный список в больших проектах.** Если активных ченджей много — список может быть длинным. Митигация: не показывать архивированные ченджи; пользователь знает своё имя ченджа и быстро найдёт его.

**R2: Поведенческое ограничение, не техническое.** Scope-ограничение в contradiction — инструкция в тексте, не code-level guard. Митигация: формулировка MUST NOT + test-кейс, проверяющий что contradiction не трогает файлы вне change-директории.
