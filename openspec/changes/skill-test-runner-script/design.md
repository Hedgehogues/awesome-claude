## Context

`skill:test-skill` — AI-скилл, который тестирует другие скиллы: разбирает `cases/<skill>.md`, материализует стабы, запускает скилл в агенте, проверяет вывод. Проблема: AI-исполнитель может пропустить ��аги (шаг 5 `status.json` был пропущен в реальной сессии). Нет внешнего enforcer'а — аналогично тому, как `openspec-apply-change` CLI обеспечивает выполнение всех задач из `tasks.md`.

Текущий инструментарий: `openspec` CLI (`/opt/homebrew/bin/openspec`) решает аналогичную проблему для apply-workflow. Нужен аналог для test-workflow.

## Goals / Non-Goals

**Goals:**
- Детерминированное выполнение всех шагов `skill:test-skill` без пропусков
- Материализация стабов (git, files, mock_commands, env) как в текущем skill.md
- Запуск скилла через `claude -p` и захват вывода
- Проверка `contains:` правил (автоматически)
- Создание и обновление `status.json` на каждом шаге
- Итоговый отчёт в том же формате, что и AI-скилл

**Non-Goals:**
- Замена AI-скилла (оба сосуществуют)
- Автоматическая проверка `semantic:` правил (требует LLM — остаётся на AI)
- Реализация нового формата кейсов или стабов (используем существующий)

## Decisions

### D1: Python, не bash

Python выбран для материализации стабов — нужен YAML-парсинг (`pyyaml`), работа с файловой системой, subprocess. Bash потребовал бы внешних зависимостей (`yq`, `jq`) и менее надёжен при ошибках.

Альтернатива: bash + yq → отвергнута из-за хрупкости и дополнительных зависимостей.

### D2: `claude -p` для запуска скилла

Скилл запускается как `claude -p "<skill-body-prompt>" --cwd <TMP>`, вывод захватывается через stdout. Это точный аналог Agent tool из AI-скилла.

Альтернатива: прямая интеграция с API Anthropic → отвергнута, усложняет установку и управление auth.

### D3: Semantic-проверки остаются на AI

`contains:` проверяются скриптом (substring match). `semantic:` требуют рассуждений — скрипт передаёт вывод агента обратно в `skill:test-skill` для оценки, либо помечает как `MANUAL`.

### D4: Расположение скрипта

`skills/skill/test-skill/scripts/run_test.py` — рядом с `skill.md` и `stubs/`. Зеркалируется в `.claude/skills/skill/test-skill/scripts/`.

### D5: YAML state в `~/.cache/awesome-claude/` — отклонение от skill.md

`skill.md` использует `trap "rm -rf '$RUN_ROOT'" EXIT` — всё удаляется автоматически (тип А, эфемерный). `run_test.py` намеренно отклоняется: state хранится в `~/.cache/awesome-claude/<run-id>/state.yaml` и не удаляется автоматически (тип Б, персистентный). Причина: возможность resume при прерывании — ценность, которую тип А не даёт.

Остальная логика (cleanup `$TMP` каждого кейса, `--clean` флаг) аналогична `--keep-tmp` из `skill.md`.

State хранится в `~/.cache/awesome-claude/<run-id>/state.yaml`. Причины:
- Персистентен между прерываниями (тип Б), не удаляется автоматически
- Не засоряет проект (`openspec/changes/` — для артефактов, не для runtime state)
- YAML выбран над JSON: человекочитаем при отладке, поддерживает комментарии, `pyyaml` уже в зависимостях

Альтернативы: `/tmp/` (тип А, не восстановим) и `openspec/changes/` (засоряет проект) — отвергнуты.

### D6: Resume через `--run-id`

Скрипт принимает опциональный `--run-id`. Если передан и `state.yaml` существует — пропускает `done`/`failed` items и продолжает с первого `pending`. Без `--run-id` — генерирует новый ID через `uuid4()[:8]`.

## Risks / Trade-offs

- `claude -p` недоступен в среде → скрипт завершается с ошибкой и сообщением об установке CLI. Митигация: проверка наличия `claude` в начале.
- Semantic-правила не проверяются автоматически → помечаются как `MANUAL`, требуют ручного прогона AI-скилла. Митигация: явное отображение в отчёте.
- Расхождение формата стабов при будущих изменениях �� скрипт и skill.md могут разойтись. Митигация: единый источник схемы стабов, тест на парсинг.
