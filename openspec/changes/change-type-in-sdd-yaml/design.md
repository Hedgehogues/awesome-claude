## Context

`.sdd.yaml` описывает change через `creates`, `merges-into` и `owner`, но не фиксирует природу самого change'а. Баги и фичи существуют в одном пространстве без различия.

Разница не только семантическая: bug-fix имеет контракт «поведение сломано → исправить», у него есть регрессионный след и приоритет «нужно сейчас». Feature — «нового поведения нет → добавить», приоритет определяется продуктом. Это влияет на описание `## Why`, на критерии приёмки, и потенциально на рендер в `sdd:apply` и `sdd:archive`.

Сейчас тип никак не зафиксирован — ни в файловой структуре, ни в скиллах.

## Goals / Non-Goals

**Goals:**
- Поле `type: bug | feature` присутствует в `.sdd.yaml` у каждого нового change'а
- `sdd:propose` явно запрашивает тип и записывает его
- Существующие ченджи без поля `type` остаются рабочими (backward compatible, default = `feature`)

**Non-Goals:**
- Изменение поведения `sdd:apply`, `sdd:archive`, `sdd:contradiction` в зависимости от типа — это downstream, отдельные ченджи
- Расширение типов за пределы `bug | feature` в этой итерации
- Валидация типа при `sdd:apply` (не блокирует, только данные)
- Автоматическое определение типа по содержимому proposal.md

## Decisions

**D1: Поле верхнего уровня в `.sdd.yaml`, не атрибут capability.**

`type` описывает сам change, а не его capability. Расположение — рядом с `owner`, не внутри `creates`.

**D2: Допустимые значения — `bug` и `feature`, без extension point.**

Два значения покрывают наблюдаемые случаи. Расширение (`chore`, `refactor`) — отдельный change с явным обоснованием. Открытый enum усложняет downstream-обработку без понятной пользы сейчас.

**D3: Default `feature` при отсутствии поля.**

Большинство существующих ченджей — фичи. Default не ломает backward compatibility; читающий код всегда получает валидное значение.

**D4: `sdd:propose` — обязательный вопрос, не опциональный.**

Тип запрашивается у пользователя при каждом `sdd:propose`. Пропустить нельзя — иначе поле останется пустым у большинства ченджей. Задать неверно — легко исправить вручную.

## Risks / Trade-offs

**R1: Неясность downstream-влияния на момент proposal.**
Как тип будет использоваться в `sdd:apply`, `sdd:archive` и рендере — не определено. Риск: поле окажется orphan-данными. Митигация: сам факт наличия явного типа в схеме создаёт точку входа для следующих ченджей; при этом ничего не ломает.

**R2: Субъективность классификации.**
Граница между «баг» и «фича» не всегда очевидна (регрессия в UX — это баг или фича-fix?). Митигация: классификация остаётся на усмотрение автора; система не валидирует «правильность» типа.
