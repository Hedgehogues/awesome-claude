## Context

awesome-claude — репозиторий скиллов для Claude Code. Используется в двух режимах:

**Dev mode**: разработчик редактирует сам awesome-claude. Сейчас `skills/` (исходники) и `.claude/skills/` (то, что видит Claude Code) — две отдельные копии. После каждого изменения нужно вручную копировать файлы. Это главная боль.

**User mode**: пользователь запускает `install.sh` в своём проекте → файлы копируются в `.claude/`. Уже работает.

Когда Клод открывает эту репу — он не знает в каком режиме работать, какие скиллы использовать, что делать первым делом.

## Goals / Non-Goals

**Goals:**
- Dev mode: одна команда (`skill:setup`) настраивает симлинки, после чего правки в `skills/` сразу видны Claude Code
- Явные точки входа: `skill:setup` для dev mode; для user mode — Claude выполняет git clone + cp напрямую
- `skill:onboarding` — единый guided entry point для Клода в этой репе
- README `## Init` — инструкция для Клода: определить режим, спросить пользователя
- Test-спеки для всех скиллов `sdd:` namespace в `skill/cases/sdd/`

**Non-Goals:**
- Автоматическая синхронизация при git pull
- Поддержка Windows
- Изменение поведения `sdd:` скиллов

## Decisions

### D1: Симлинки, не watchers
Можно следить за изменениями через `fswatch` / `entr` и копировать. Но симлинки проще, не требуют фонового процесса и работают нативно на macOS/Linux. `ln -sfn $(pwd)/skills .claude/skills` — одна команда, нет зависимостей.

### D2: `skill:setup` — единственная точка входа для dev mode
`skill:setup` — Claude-way интерфейс: выполняет bash-команды создания симлинков напрямую, без делегирования к внешнему скрипту. Нет зависимости от Makefile или внешних инструментов.

### D3: Симлинки на уровне namespace, не файлов
`ln -sfn $(pwd)/skills .claude/skills` заменяет директорию целиком. При добавлении нового файла в `skills/` он автоматически появляется в `.claude/skills/` — не нужно ничего делать. После симлинка `skills/skill` автоматически виден как `.claude/skills/skill`.

### D4: `skill:onboarding` — информационный, не исполняемый
Скилл проверяет наличие симлинков и список активных ченжей, показывает статус и следующий шаг. Не запускает другие скиллы автоматически. Причина: guided workflow не должен захватывать управление; Клод может быть на разных стадиях.

### D5: README `## Init` — инструкция для Клода, не для человека
Клод читает README при открытии репы. Раздел `## Init` содержит явную инструкцию: проверить директорию, предложить режим, спросить пользователя. Не нужен отдельный CLAUDE.md для этой логики — README достаточно.

### D6: `manifest.yaml` — единый источник правды для зависимостей
Версии CLI-инструментов сейчас не формализованы в файле; субмодули — в `manifest.yaml repos:` (управляются `sdd:repo`), но самого `manifest.yaml` в репе ещё нет. Решение: ввести `manifest.yaml` в корне с тремя секциями — `version:` (версия репы), `tools:` (версии CLI-инструментов), `repos:` (субмодули). `skill:deps` читает только `manifest.yaml`. `sdd:repo` продолжает управлять `repos:` без изменений.

### D7: Определение режима по директории
Если Клод запущен в директории самой awesome-claude репы (есть `skills/`, `commands/`, `manifest.yaml`) — предлагать dev mode. В любой другой директории — user mode.

### D8: Удалить install.sh — Claude выполняет git clone + cp напрямую
`scripts/install.sh` удаляется. Причина: curl-pipe-bash — небезопасный и непрозрачный паттерн. В user mode Claude сам выполняет git clone + cp напрямую, без внешних скриптов. README не документирует curl | bash.

### D9: Eval-фреймворк для test-спеков — LLM-judge + bootstrap
Test-спеки проверяют скиллы через assertions двух типов: `contains:` (детерминированные) и `semantic:` (оценивает LLM-judge отдельным вызовом: "удовлетворяет ли вывод критерию? yes/no + причина"). Каждый кейс прогоняется k=5 раз (bootstrap); засчитывается при ≥4/5. Прогресс выводится в реальном времени (`[case] 1/5 ✓ 2/5 ✓ …`), при провале — `⚠ WARN`. Файл результатов создаётся в начале прогона (`test-results/YYYY-MM-DD-HHMMSS.md`), каждый результат пишется по мере выполнения.

### D10: Переименование namespace `__dev` → `skill`
*Infrastructural decision — no spec required (rename директорий и frontmatter, не функциональный контракт).*

В репе есть namespace `__dev` для dev-инфраструктуры (`__dev:test-skill`, `__dev:test-all`). Префикс `__` — внутренняя метка «не для пользователя», но смысл namespace — «инструменты для контрибьютора», что точнее покрывается именем `skill`. Этот change вводит ещё четыре скилла того же класса (`skill:setup`, `skill:deps`, `skill:onboarding`, `skill:release`). Решение: переименовать `__dev` → `skill` целиком (директории `skills/__dev/` → `skills/skill/`, `commands/__dev/` → `commands/skill/`, frontmatter `name:` во всех файлах). Альтернатива (оставить `__dev` для test-инфраструктуры, новые скиллы — в `skill`) отклонена: два namespace для одного класса инструментов размывают границу.

### D11: `skill:deps` — единая точка оркестрации; read-only по `manifest.yaml`
Установка зависимостей и инициализация субмодулей — две операции, которые контрибьютор всегда делает вместе. Решение: один скилл-оркестратор `skill:deps` читает `manifest.yaml`, устанавливает инструменты из `tools:` и вызывает `sdd:sync` для `repos:`. Альтернатива (встроить в `skill:setup`) отклонена: setup — про симлинки (быстро, идемпотентно), deps — про сеть и подмодули (долго, может частично падать). Разные failure-режимы → разные скиллы. Read-only-контракт по `manifest.yaml` см. `specs/deps-skill/spec.md` → Requirement: skill:deps does not modify manifest.yaml. Rationale: защита от silent-state-changes при запуске инсталлятора; добавление/удаление записей — ручным редактированием или через `sdd:repo` для `repos:`.

### D12: Schema `manifest.yaml` фиксирована и валидируется skill:deps
`manifest.yaml` имеет строго три top-level ключа: `version`, `tools`, `repos`. Записи вне этих ключей невалидны. `skill:deps` валидирует наличие всех трёх ключей при чтении и останавливается с ошибкой, если хотя бы одного нет. Альтернатива (валидация через отдельный CLI-инструмент) отклонена: лишняя зависимость, и contributor всё равно вызывает `skill:deps` первым делом.

### D13: Test-спеки покрывают все sdd-скиллы по фиксированному паттерну
Покрытие тестами для sdd-namespace должно быть полным и явным: каждому скиллу `sdd:<name>` соответствует ровно один test-spec по фиксированному пути `skills/skill/cases/sdd/<name>.md`. Альтернатива (выборочное покрытие или плоский список в одном файле) отклонена: фиксированный путь делает наличие test-spec'а легко проверяемым (`ls skills/skill/cases/sdd/`) и связывает имя файла с именем скилла без дополнительной индексации.

## Risks / Trade-offs

[Конфликт симлинк / существующая директория] → `skill:setup` проверяет, является ли `.claude/skills/` уже симлинком. Если нет — выводит предупреждение и просит подтверждение перед заменой.

[Сломанные симлинки после `git clean`] → `skill:setup` нужно перезапустить. Документируется в README.

[`skill:setup` без внешних зависимостей] → скилл выполняет bash-команды напрямую; нет внешних скриптов, нет зависимостей от файлов вне `skills/skill/setup.md`.

[Тест-спеки для `sdd:apply` требуют ченжа в стабе] → новый стаб `with-change` включает `openspec.changes`; если openspec CLI недоступен в стабе — кейс SKIP.
