## Context

`bump-namespace.sh` — скрипт, который пользователи запускают при обновлении неймспейса. Сейчас он:
- работает только с тегами (контрибьютор не может тестировать ветку до релиза)
- копирует только `skills/<namespace>/` — рулы, скрипты и другие файлы не доезжают
- не знает структуру неймспейса, пути захардкожены
- не оставляет точки входа для Claude в проекте пользователя

## Goals / Non-Goals

**Goals:**
- Дать контрибьютору возможность тестировать изменения с ветки через `--ref`
- Гарантировать полную доставку неймспейса через `namespaces:` в `.manifest`
- Оставить Claude точку входа в проекте пользователя через `CLAUDE.md` + `index.md`
- Устойчивый формат `.versions` без null-состояний при любом типе ref

**Non-Goals:**
- Переход к следующему dev-циклу (это `skill:post-release`, отдельный change)
- Изменение логики `skill:release`
- Поддержка pre-release меток (alpha, rc)

## Decisions

**1. Формат `.versions`**

`.versions` хранит тип + лейбл + разрешённый SHA через `@`:

```
dev=tag:v0.7.0@a3f9c12
dev=branch:release-0.7.0@a3f9c12
dev=sha:a3f9c12
```

Логика «уже установлен» сравнивает только SHA (часть после `@`, или всё значение если формат `sha:`). Тип и лейбл — только для читаемости человеком, в сравнении не участвуют.

При установке с ветки скрипт резолвит HEAD ветки в SHA в момент клонирования и записывает его. При следующем запуске — снова резолвит HEAD и сравнивает SHA: обновляет только если SHA изменился. Null-состояний нет: любой тип ref даёт один и тот же формат записи.

**2. Точка входа для Claude в `CLAUDE.md` пользователя**

`bump-namespace.sh` при первой установке любого неймспейса добавляет в `CLAUDE.md` пользователя раздел:

```markdown
## awesome-claude

Source: https://github.com/Hedgehogues/awesome-claude
Index:  .claude/awesome-claude/index.md
```

`index.md` приезжает с bump и содержит реестр всего установленного — скиллы, рулы, скрипты. Claude подцепляет его как точку входа. В `CLAUDE.md` только ссылка, детали не дублируются. Неустановленные неймспейсы в `index.md` не попадают.

**3. Поддержка `--ref` в `bump-namespace.sh`**

Скрипт принимает необязательный флаг `--ref <tag|branch|sha>`. Без флага — поведение как сейчас (последний тег). С флагом — клонирует по указанному ref, печатает WARNING в stdout, зависимости не обновляет автоматически.

Это закрывает кейс контрибьютора: протестировать изменения с ветки до создания релизного тега.

**4. Поле `namespaces:` в `.manifest` каждого неймспейса**

`.manifest` расширяется полем `namespaces:` — список всего, что принадлежит неймспейсу:

```yaml
depends_on: [opsx]
namespaces:
  - skills/dev/
  - rules/dev/
  - scripts/dev/
```

`bump-namespace.sh` читает `namespaces:` и копирует все перечисленные папки. Это гарантирует что пользователь получает весь неймспейс целиком — не только скиллы, но и рулы, скрипты и всё остальное.

## Risks / Trade-offs

- [Risk] `.manifest` у пользователя без поля `namespaces:` — при обновлении скрипт упадёт с ошибкой → Mitigation: bump-namespace.sh проверяет поле и выдаёт понятное сообщение.
- [Risk] Ветка нестабильна — пользователь устанавливает с ветки, завтра она двигается → Mitigation: WARNING в stdout при `--ref` не-тег; зависимости не обновляются автоматически.
- [Risk] `CLAUDE.md` пользователя не существует → Mitigation: скрипт создаёт его с минимальным содержимым.
- [Trade-off] `namespace-first-structure` (активный change) планирует реструктуризацию путей → наши решения по `namespaces:` и путям придётся пересмотреть после реализации того change'а; зафиксировано в `namespace-first-structure/design.md`.
