## Context

Мотивация и описание текущего состояния — см. `proposal.md → ## Why`.

Оставшийся пробел: скрипты вызываются через `Bash tool` с позиционными аргументами и текстовым выводом без типизации.

MCP (Model Context Protocol) — протокол Claude для работы с внешними инструментами через декларативный, типизированный интерфейс. Инструменты MCP имеют JSON-schema на входе, структурированный ответ и версионирование. Вызов через MCP заменяет прямой `bash <script> <args>`.

Текущее состояние: MCP-сервер уже используется в проекте, но правило о том, что скрипты данных должны проходить через MCP, отсутствует.

## Goals / Non-Goals

**Goals:**
- Ввести принцип: взаимодействие с внешними системами — через MCP-интерфейс; начальный scope и условия расширения определяются инкрементально
- Определить поведение Claude при отсутствии MCP-интерфейса: предложить реализацию пользователю
- Закрепить правило в `.claude/rules/mcp-script-access.md` (отдельный файл, не правка `claude-way.md`)

**Non-Goals:**
- Немедленно мигрировать существующие скрипты на MCP — это отдельная работа
- Определять конкретную технологию MCP-сервера (FastMCP, mcp-python и т. д.)
- Отменять правило о Python-скриптах — MCP-тул обёртывает скрипт, не заменяет его

## Decisions

**D1: Отдельный файл правила, не правка `claude-way.md`.**

`claude-way.md` описывает архитектурный принцип («что»), `mcp-script-access.md` описывает процедурное требование («как именно Claude взаимодействует со скриптами»). Разделение облегчает независимое обновление каждого из принципов.

**D2: Поведение при отсутствии MCP — предложить, а не жёстко блокировать.**

Claude должен предложить реализацию MCP-интерфейса, но не применять break-stop. Скрипты без MCP могут существовать в legacy-контексте или при первичной разработке. Если пользователь явно разрешает прямой вызов — Claude продолжает с явной пометкой об отступлении от правила.

**D3: Scope правила открытый — расширяется по мере практики.**

Начальный scope совпадает с `claude-way.md` (структурированные данные: YAML, JSON, TOML, INI, CSV, БД). Распространение на сервисы, HTTP API, CLI-инструменты и другие внешние системы — инкрементально, по мере того как становится ясно, где MCP-интерфейс реально нужен и работает. Скрипты без схемы (Markdown, free-text) — вне scope на любом этапе.

**D4: Обновление `claude-way.md` — forward-pointer, не переработка.**

Секция «Структурированные данные» в `claude-way.md` дополняется одной строкой: «точка входа — MCP, см. `mcp-script-access.md`». Содержательная часть правила не переписывается — только добавляется ссылка на новый файл. Причина: `claude-way.md` — shared document, минимальный diff снижает риск регрессий в других проектах, использующих его.

## Risks / Trade-offs

**R1: Трение при работе с существующими скриптами без MCP-интерфейса.**

Каждый новый вызов потребует либо создания MCP-тула, либо явного разрешения пользователя. Митигация: правило описывает желаемое состояние, а не мгновенную блокировку — переход инкрементальный по мере того, как скрипты обрастают MCP-интерфейсами.

**R2: Размытие ответственности между `claude-way.md` и новым правилом.**

Обе нормы упоминают структурированные данные, что может создать путаницу о приоритете. Митигация: в `claude-way.md` добавить явную ссылку: «точка входа — MCP, см. `mcp-script-access.md`».
