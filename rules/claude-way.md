# Claude-way

## Единственный интерфейс

Пользователь взаимодействует с awesome-claude только через Claude Code. Прямые системные вызовы (`bash`, `curl` и т.д.) не являются частью публичного API — они внутренняя реализация. Любая операция (установка, обновление, запуск скилла) выполняется через Claude Code.

## Гранулярность скилла

Последовательность скилл-вызовов является bounded context'ом — она определяет его границу. Скилл организует один шаг внутри этой последовательности.

- Шагов должно быть мало: **1–2 предпочтительно, не более 3**
- Артефакт существует внутри своего шага и не переходит в следующий как общий объект
- Следующий шаг принимает выходы одного или нескольких предыдущих шагов как свои собственные входные артефакты
- Каноническая форма — **два этапа по аналогии с ETL**: первый шаг производит сырой артефакт, второй преобразует его в итоговый

Длинная последовательность (> 3 шагов) или разделяемые артефакты между шагами — сигнал к пересмотру границ.

## Жизненный цикл артефакта

Каждый артефакт, создаваемый или изменяемый скиллом, декларирует:

1. **Область ответственности** — что он описывает и что намеренно не описывает
2. **Время жизни**:
   - `ephemeral` — существует только в рамках одного вызова
   - `session` — существует на протяжении сессии Claude Code
   - `persistent` — записывается на диск и переживает сессию

Артефакт без явных границ нарушает принцип bounded context'а.

## Структурированные данные

Любой доступ (чтение и запись) к данным со схемой или повторяющейся структурой выполняется через Python-скрипт, не нативно.

**В scope:** YAML, JSON, TOML, INI, CSV, базы данных, `.versions`, `.manifest`

**Вне scope:** Markdown, исходный код, free-text конфиги без схемы

Lifetime скрипта соответствует lifetime трансформации: одноразовая операция — `ephemeral`, регулярная — `persistent`.
