# Domain

## Контекст
Домен — это зафиксированный смысл бизнеса и набор правил, по которым система
должна принимать решения. Этот блок объясняет, как описывать предметную область
через bounded contexts, единый язык и бизнес-возможности, чтобы избежать
разночтений между продуктом и кодом. Контекст важен для выбора агрегатов,
формализации команд/событий/запросов и построения модели данных, которая
следует за доменом, а не диктуется инфраструктурой. Ниже перечислены правила,
которые служат чеклистом при анализе, моделировании и валидации доменной модели.

## Правила описания домена
- Описывать домен через bounded contexts. Делай карту контекстов: названия,
  границы ответственности, интеграции. Один контекст — одна модель.
- Фиксировать ubiquitous language. Заведи глоссарий терминов и запрети
  синонимы. Если в коде “Agent” и “Bot” — это либо разные сущности, либо бардак.
- Начинать с бизнес-возможностей (capabilities). Для каждого capability: цель,
  акторы, входы/выходы, правила, метрики успеха.
- Определять агрегаты и их инварианты. Для каждого агрегата: root, список
  сущностей/VO внутри, инварианты, командами как меняется, событиями что
  фиксируется.
- Выделять сущности и value objects.
- Entity: identity, жизненный цикл.
- Value Object: без identity, неизменяемый, сравнение по значению.
- Формализовать команды, события, запросы.
- Команды: намерения, вход в write слой.
- События: факты после коммита.
- Queries: чтение через проекции/витрины.
- Описывать бизнес-правила как политики/спеки. Правила должны быть тестируемыми
  и жить в домене, а не в контроллере.
- Использовать доменные сервисы только когда логика не принадлежит агрегату.
  Если можно запихнуть в агрегат — запихни. Сервисы — для “между объектами”,
  не для ленивых.
- Процессы между агрегатами оформлять явно. Saga/Process Manager: состояния,
  переходы, таймауты, компенсации. Никаких “ну мы просто в одном методе вызовем
  два репо”.
- Интеграции описывать контрактами. API/события: схемы, версии, гарантии
  доставки, идемпотентность, порядок.
- Делать модель данных производной от доменной модели. Таблицы/коллекции
  отражают агрегаты и их границы. Если БД диктует домен — ты строишь не домен,
  а культ SQL.
- Формализовать ошибки домена. Код ошибки, причина, контекст. Ошибки — часть
  модели и контрактов.
- Встроить контроль качества в процесс. Любое нарушение границ (агрегаты,
  транзакции, доступы) → задача на техдолг с ссылкой на правило. Без “потом”.
