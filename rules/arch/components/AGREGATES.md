# Aggregates

## Контекст
Агрегат — транзакционная и смысловая граница домена, в которой держатся
инварианты и принимаются решения о допустимых изменениях состояния. Этот раздел
фиксирует, как выбирать размер агрегата, через какие входы он меняется, и какие
связи допустимы с другими агрегатами. Контекст нужен, чтобы согласовать
поведение доменной модели, репозиториев и данных хранения, а также предотвратить
скрытую связность и транзакции «через весь домен». Все последующие правила
читаются как ограничения на чтение/запись и координацию между компонентами.

## Правила
- Агрегаты не пересекаются. Один инвариант — один хозяин. Попытка «поделиться
  полями» = будущий пожар.
- Коммит атомарный. Агрегат либо целиком записался в БД, либо целиком откатился.
  Полумеры — для слабых.
- Связи между агрегатами — только по первичным ключам. Никаких ссылок на
  внутренние поля, коллекции и «дай мне кусочек потрохов».
- Никакого прямого доступа к внутренностям чужого агрегата. Хотите данные —
  берите через ID и отдельный запрос/проекцию.
- Модификация агрегата — только через репозиторий. В обход — это не «быстрее»,
  это «техдолг с процентами».
- Загрузка агрегатов — через репозитории + lazy loading, где уместно. Не тащим
  весь мир в память ради одной строчки.
- Агрегат в памяти консистентен. Либо загружен целиком, либо догружается лениво
  так, чтобы инварианты не разваливались.
- Чтение — отдельный query/view слой. Команды меняют, запросы читают. Не мешаем
  суп с компотом.
- CQRS — логическое разделение на уровне сервиса, без фанатизма про
  масштабирование. Просто чтобы не плодить монстров из
  “one method to rule them all”.
- Модели данных отражают границы агрегатов. Таблицы/документы не должны
  «склеивать» два агрегата в одного мутанта.
- Модели данных повторяют требования, а не фантазии разработчика. Если
  расходится — правим код, не реальность.
- Агрегаты версионируются. Optimistic locking/версия — чтобы конкуренция не
  превращалась в рандом.
- Нарушение правил = задача на рефакторинг/техдолг. Не “потом как-нибудь”, а
  фиксируем и выпиливаем постепенно, без героизма и ночных дежурств.
- У агрегата есть один корень (Aggregate Root). Внешний мир взаимодействует
  только с ним. Остальное — внутренняя кухня.
- Все инварианты агрегата обеспечиваются только внутри агрегата. Ни сервисы,
  ни контроллеры, ни «временные хаки» не имеют права поддерживать
  консистентность.
- Транзакции не выходят за пределы одного агрегата. Межагрегатная
  консистентность — через eventual consistency, события или саги.
- Агрегаты не используют репозитории других агрегатов. Максимум — работа с ID
  и доменными событиями.
- Доменные события публикуются после успешного коммита агрегата. Не до, не
  «где-то в середине», а строго после.
- Один репозиторий — один агрегат. Репозиторий не знает и не управляет чужими
  агрегатами.
- Бизнес-логика живёт в агрегатах, не в сервисах. Сервисы оркестрируют,
  агрегаты принимают решения.
- Агрегат не зависит от инфраструктуры. Никаких ORM, HTTP, Kafka и прочей
  цивилизации внутри доменной модели.
- Размер агрегата минимален. Всё, что не нужно для инвариантов, выносится за
  его пределы. Большие агрегаты — признак архитектурной лени.
- Создание агрегата — через фабрики или конструкторы с валидным состоянием.
  Агрегат не может существовать в «полусобранном» виде.
- Удаление агрегата — явная бизнес-операция. Никаких silent delete, cascade по
  настроению и «само пропало».
- Агрегаты не шарят транзакционное состояние. Никаких shared locks,
  shared sessions и прочих костылей.

## Термины
- Агрегат: группа связанных данных и логики, которая изменяется как единое
  целое и объединяются бизнес-правилами.
- Граница транзакции: рамка атомарного изменения данных, внутри которой
  операции выполняются целиком или не выполняются вовсе.
