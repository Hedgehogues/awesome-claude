# Events

## Контекст
Доменные события фиксируют факты, которые уже произошли, и становятся основой
для интеграций и eventual consistency. Они формулируются в прошедшем времени:
UserRegistered, InvoicePaid, AgentPublished, а не “PayInvoicePlease”.
Контекст задает момент появления события (после коммита агрегата), минимальный
payload и обязательные метаданные, чтобы потребители могли корректно
идемпотентно обрабатывать факт. Эти правила отделяют доменные события от
инфраструктурных и задают дисциплину публикации и обработки.

## Правила
- Событие неизменяемо. Записали — всё, назад только новым событием
  (компенсацией).
- Событие описывает домен, а не инфраструктуру. Не KafkaMessageSent,
  а OrderShipped.
- Событие рождается внутри агрегата (обычно в Root). Снаружи агрегат не
  “подделывают”.
- Событие публикуется только после успешного коммита агрегата. Иначе ты
  разошлёшь ложь.
- Событие относится к одному агрегату-источнику. В payload — его aggregate_id,
  aggregate_type, version.
- Payload минимальный. Либо “ключи для чтения” (ID), либо небольшой снапшот
  “что важно”, но без гигантских графов объектов.
- События версионируются. event_version/schema version, чтобы не ломать
  потребителей при эволюции.
- События идемпотентно обрабатываются. Повторная доставка — норма, не баг.
- Событие имеет метаданные. event_id, occurred_at, correlation_id, causation_id,
  actor_id (если есть), tenant_id (если есть).
- Порядок событий важен внутри одного агрегата. Между агрегатами — нет
  гарантий, живём с eventual consistency.
- Межагрегатная консистентность через события/саги. Не через “одна транзакция
  на всё” (это сказки).
- Доставка событий — через outbox. События пишутся в БД вместе с агрегатом
  в одной транзакции, публикация — отдельным процессом.
- Обработчики событий не должны мутировать чужой агрегат напрямую.
  Команды/саги/оркестрация — ок; “влез и поправил поля” — нет.
- Интеграционные события отделяются от доменных. Доменные — внутрь
  bounded context, интеграционные — наружу (могут быть маппинг/адаптация).

## Типы событий
- Domain Event — внутренний факт домена (ядро).
- Integration Event — то, что отдаём наружу (стабильнее, чище, часто беднее
  по данным).
- System/Technical Event — логи/метрики/трассировка (не путать с доменом).
