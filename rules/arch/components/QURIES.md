# Queries

## Контекст
Query — это запрос на чтение без изменений состояния, поэтому любой side-effect
в query считается багом. Контекст определяет границы read-слоя, его отделение от
команд и агрегатов, а также требования к предсказуемости, производительности и
безопасности. Здесь фиксируется, что read-модель может отличаться от write-модели
и строиться под конкретные сценарии чтения, включая денормализацию и кеширование.
Эти правила помогают сохранять чистое разделение CQRS и стабильные контракты
чтения.

## Правила
- Queries отделены от команд логически (CQRS на уровне сервиса). Один метод не
  должен одновременно читать и писать.
- Чтение не обязано использовать агрегаты. Read слой не соблюдает инварианты —
  он отображает данные.
- Для чтения используются отдельные view-функции/DAO/Query Service. Репозитории
  агрегатов для чтения не дергаем “по привычке”.
- Read Model может отличаться от Write Model. Структура под скорость/удобство
  чтения, не под “как у нас в домене”.
- Read Model денормализуется по потребностям UI/кейсов. Нормализация ради
  святости — в мусорку.
- Read слой не возвращает доменные сущности. Возвращает DTO/Projection/View.
- Read Model может собираться из нескольких агрегатов. Это ок, потому что это
  не транзакционная граница.
- Консистентность read-моделей может быть eventual. Если нужна строгость —
  явный контракт и отдельный сценарий (без магии).
- Хранилище read-моделей выбирается под чтение. Postgres view/таблицы, Elastic,
  ClickHouse — что выгоднее, а не “что уже стоит”.
- Queries должны быть предсказуемыми по производительности. Ограничения:
  пагинация, лимиты, индексы, запрет на N+1.
- Queries могут кешироваться. Кеш — часть read-слоя, с явной стратегией
  инвалидции/TTL.
- Версионирование read DTO обязательно при изменении контрактов. Клиенты не
  должны страдать “внезапно”.
- Read слой имеет отдельные права доступа. Кто может читать что — не менее
  важно, чем кто может писать.
- Наблюдаемость: метрики и трассировка. latency, hit rate, slow queries,
  количество результатов.

## Обновление read-моделей
- On-demand: запросы напрямую в БД/индексы.
- Проекции по событиям: event handlers → materialized views.
