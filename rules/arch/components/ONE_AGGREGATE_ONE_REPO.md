# One Aggregate = One Repository (R10)

## Контекст

Каждый доменный поддиректории (`src/domain/{name}/`) представляет один агрегат —
транзакционную границу с единственным root entity. Правило формализует ограничение
**1 домен = 1 агрегат = 1 абстрактный репозиторий = 1 конкретная реализация**.

## Мотивация

- **Транзакционная граница**: один репозиторий = одна единица консистентности.
  Если в `repository.py` два ABC, значит два агрегата живут в одном домене,
  и их транзакции могут неявно переплетаться.
- **Навигация**: разработчик находит репозиторий по имени домена, не перебирая
  несколько ABC внутри файла.
- **Тестируемость**: моки 1:1 — один mock per repository, без неоднозначности.

## Правило

### R10a: Один абстрактный репозиторий на домен

Файл `src/domain/{name}/repository.py` ОБЯЗАН содержать ровно один класс,
наследующий от `ABC`. Этот класс — абстрактный репозиторий агрегата.

### R10b: Один файл конкретной реализации на домен

Для каждого агрегата ОБЯЗАН существовать файл
`src/infrastructure/database/{name}_repo.py`. Не допускается наличие нескольких
`*_repo.py` файлов, ссылающихся на один и тот же доменный агрегат.

### R10c: Конкретный репозиторий реализует абстрактный своего агрегата

Файл `{name}_repo.py` ОБЯЗАН импортировать абстрактный репозиторий из
`src.domain.{name}.repository`. Это гарантирует связь конкретной реализации
с правильным доменом.

## Что является нарушением

- Два и более ABC в одном `repository.py`:
  ```python
  # BAD: src/domain/integration/repository.py
  class IntegrationRepository(ABC): ...
  class PipelineRunRepository(ABC): ...  # Нарушение R10a
  ```
- Два `*_repo.py` файла, импортирующих из одного `src.domain.{name}`:
  ```
  integration_repo.py      → src.domain.integration.repository
  pipeline_run_repo.py     → src.domain.integration.repository  # Нарушение R10b
  ```

## Что НЕ является нарушением

- **Child entities** в том же `entity.py` — это нормально (R6). Child entity
  не имеет своего репозитория, она персистится через репозиторий root.
- **Несколько SQLAlchemy-моделей** для одного агрегата (`RunModel` + `AnomalyModel`) —
  это нормально. Модели маппят child entities на отдельные таблицы.
- **Вспомогательные классы** (DTO, exceptions) в `repository.py` — допустимы,
  если это не ABC-репозитории.

## Связь с другими правилами

- **R3** (Repository isolation): каждый `_repo.py` импортирует только модели
  своего агрегата. R10 дополняет это: для каждого домена ровно один такой файл.
- **R6** (Child entity co-location): child entities живут в `entity.py` root-а
  и не имеют своего репозитория.
- **AGGREGATE_STRUCTURE.md**: описывает полную структуру агрегата; R10
  автоматически проверяет соответствие 1:1:1.

## Исправление нарушений

Если обнаружено несколько ABC в одном домене, варианты:
1. **Вынести в отдельный домен**: создать `src/domain/{child_name}/` с собственным
   entity, repository, repo.
2. **Сделать child entity**: если сущность не имеет независимого жизненного цикла,
   управлять ей через репозиторий root-агрегата (добавить методы в существующий ABC).
