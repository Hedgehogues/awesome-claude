# Write-модель строится от агрегатов. Сначала агрегаты/границы, потом таблицы. Не наоборот.

# Контекст
Документ фиксирует правила проектирования write-модели и границ агрегатов. Используется как базовый ориентир для структуры таблиц и инвариантов записи.


## Правила

- Каждый агрегат имеет root-таблицу. Одна строка = одно состояние root. Остальное — дочерние таблицы.
- Транзакция записи всегда ограничена одним агрегатом. В одной транзакции не трогаем два агрегата. Никогда.
- Связи между агрегатами — только по первичным ключам. В таблицах хранится *_id, без JOIN’ов “за инвариантами”.
- FK между агрегатами допускаются только как ссылочная целостность, не как механизм доменной логики. Никаких каскадов “чтоб само”.
- Внутри агрегата допускаются FK и каскады. Только внутри границы агрегата.
- Инварианты агрегата фиксируются constraints’ами. NOT NULL, CHECK, UNIQUE, FK — где возможно, без “проверим в коде”.
- У root-таблицы агрегата обязательно поле версии. version / row_version для optimistic locking.
- Любая запись агрегата — атомарный коммит. Агрегат либо записан целиком, либо откат целиком.
- Запись агрегата выполняется через репозиторий агрегата. Прямой SQL “в обход” — нарушение.
- Один репозиторий ↔ один агрегат. Репозиторий не пишет в таблицы другого агрегата.
- Структура таблиц агрегата не отражает view-кейсы. Никаких денормов “для удобного экрана” в write-модели.
- Дублирование данных в write-модели запрещено по умолчанию. Исключения — только с обоснованием (perf/ограничения) и документируются.
- 3НФ по умолчанию для write-модели. Денормализация — только осознанно и точечно.
- Все коллекции внутри агрегата — отдельные таблицы. Не JSON-свалка, если это часть бизнес-инвариантов.
- JSON/JSONB разрешён только для “схема-нестабильных” атрибутов. И только если на них нет критичных инвариантов/уникальностей.
- Имена таблиц/полей отражают ubiquitous language. Не tbl1, не data, не “как в ORM назвалось”.
- Идентификаторы стабильны и неизменяемы. id не переиспользуется, не меняется, не “перегенерим”.
- Мягкое удаление (soft delete) — только по доменному требованию. Иначе — нормальное удаление + ретеншн/архив.
- Аудит изменений фиксируется явно. created_at, updated_at, created_by, updated_by — если требуется доменом/комплаенсом.
- Мульти-тенантность фиксируется на уровне схемы. tenant_id в root и дочерних таблицах (если применимо), правила изоляции документируются.
- Уникальности и бизнес-ключи определяются на уровне БД. Никаких “в коде проверим, что email не занят”.
- Запрет на cross-aggregate JOIN при записи. Любая логика “проверить чужое состояние и записать” решается через события/саги/компенсации.
- Outbox обязателен для публикации доменных/интеграционных событий. События пишутся в одной транзакции с агрегатом.
- Все write-операции идемпотентны, где есть внешние повторы. Используем idempotency_key/уникальные ограничения.
- Миграции write-модели делаются безопасно (expand/contract). Никаких “добавили NOT NULL и пошли пить чай”.
- Индексы в write-модели — только под записи/констрейнты. Под чтение — в read-модели.
