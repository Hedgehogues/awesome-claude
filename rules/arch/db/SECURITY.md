# SECURITY.md — правила безопасности данных

# Контекст
Документ фиксирует правила безопасности доступа к данным, роли БД и работу с PII. Определяет базовые требования к изоляции и аудиту.

## Общие принципы
- Безопасность — часть архитектуры, а не middleware.
- Если надеешься “проверим в контроллере” — ты уже проиграл.
- Минимальные привилегии по умолчанию.
- Любой доступ нужно обосновать. Всё лишнее запрещено.
- Write и Read доступы разделены.
- Кто пишет ≠ кто читает. Даже если сейчас “один сервис”.

## Доступ к БД
- Отдельные роли БД для:
  - write-пути
  - read-пути
  - миграций
  - админских операций
- Приложение не подключается под superuser.
- Никогда. Даже “на тесте”.
- DDL запрещён из runtime-кода.
- Схема меняется только миграциями.

## Изоляция данных (multi-tenant / контексты)
- Tenant isolation формализуется.
- Один из вариантов выбирается и фиксируется:
  - tenant_id во всех root-таблицах
  - схемы БД на tenant
  - отдельные БД
- Правила изоляции одинаковы для write и read.
- Нет “в read можно схалтурить”.
- Row Level Security (RLS) используется при необходимости.
- Политики документируются, тестируются, не “на глаз”.

## PII и чувствительные данные
- PII минимизируется.
- Не храним то, что не нужно домену.
- PII помечается явно.
- В схеме/документации: какие поля чувствительные.
- Шифрование:
  - at rest — для чувствительных полей
  - in transit — всегда
- Read-модели не дублируют PII без необходимости.
- Если дублируют — маскирование обязательно.
- Удаление PII соответствует политике ретенции.
- deleted_at ≠ удалено навсегда.

## Аутентификация и авторизация
- Авторизация не зашивается в SQL-логику хаотично.
- Явные правила: где RBAC, где ABAC, где tenant-фильтры.
- Решения о доступе принимаются до write-транзакции.
- Не после, не “в середине”.
- Read-доступ проверяется отдельно.
- “Можно читать” ≠ “можно писать”.

## Read-модели и утечки
- Read-модели рассматриваются как публичная поверхность.
- Всё, что туда попало, считается потенциально утёкшим.
- Проекции фильтруются по доступам.
- Не “вернём всё, а фронт скроет”.
- Поиск и сортировки не должны раскрывать скрытые данные.
- Метаданные тоже считаются данными.

## Аудит и наблюдаемость
- Критичные операции логируются.
- Кто, когда, над каким агрегатом, что сделал.
- Аудит не нарушает инварианты.
- Логи ≠ бизнес-логика.
- Доступ к аудит-логам ограничен.
- Это тоже чувствительные данные.

## Инциденты и ошибки
- Ошибки не раскрывают внутреннюю структуру БД.
- Никаких SQL, stack trace и имён таблиц наружу.
- Подозрительная активность алертится.
- Повторы команд, конфликты версий, ошибки доступа.

## Антипаттерны (запрещено)
- Один пользователь БД “на всё”
- Проверка прав только на уровне UI
- PII в read-моделях “потому что удобно”
- Логи с полными payload’ами команд/событий
- Shared credentials между сервисами
