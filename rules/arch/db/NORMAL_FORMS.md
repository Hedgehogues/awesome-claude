# NORMAL_FORMS.md — правила нормализации (где да/где нет)

# Контекст
Документ описывает требования к нормализации write-модели и границы допустимой денормализации. Помогает сохранять целостность данных без ухудшения читаемости и производительности.


## Правила

- Нормализация применяется только к write-модели. Read-модель — отдельная история, там денорм рулит.
- Write-модель по умолчанию в 3НФ. Это базовая страховка от дублей, рассинхрона и “почему цифры разные”.
- Денормализация в write-модели запрещена по умолчанию. Разрешается только по документированному исключению.
- Каждое исключение обязано иметь причину и доказательство. Причина = измеримая производительность/ограничение/стоимость, доказательство = план запроса/нагрузочный профиль/метрика.
- Любая денормализация в write-модели должна иметь источник истины. Одно поле “главное”, остальные — кэш/копия и это явно отмечено.
- Дубли в write-модели должны быть автоматически поддерживаемыми. Триггеры/проекции/код в репозитории — но не “человек помнит”.
- Нормализация не должна ломать границы агрегатов. Никаких “вынесем общее в shared table” если это склеивает агрегаты.
- Таблица не должна содержать поля из разных агрегатов. Если содержит — это не “нормализация”, это “архитектурный брак”.
- Для справочников/каталогов используется отдельный слой reference data. Не размазывать константы по всем таблицам.
- Многие-ко-многим внутри агрегата допустимы. Между агрегатами — только через ID и отдельные механики (события/саги), не через общую join-таблицу как “общий желудок”.
- Повторяющиеся атрибуты → value object таблицы или embedded-структура внутри агрегата. Выбор зависит от инвариантов и частоты изменений.
- Сильно “схема-нестабильные” атрибуты можно хранить в JSON/JSONB. Но:
  - без критичных инвариантов на этих полях
  - без уникальностей
  - без ключевых фильтров в hot-path
- Ключевые фильтры и сортировки не должны жить в JSON. Иначе ты строишь БД-лабиринт.
- Уникальности (business keys) фиксируются в БД. В 3НФ это обязано быть constraint’ом, а не “мы проверим перед вставкой”.
- Аномалии обновления недопустимы. Если изменение требует править 5 строк в 5 местах — это не модель, это наказание.
- Нормализация должна помогать целостности, а не мешать производительности. Если 3НФ убивает latency — переносим чтение в read-модель, а не калечим write.

## Политика “где 3НФ, где денорм”
- Write model (агрегаты/репозитории): 3НФ, минимум дублей, строгие constraints.
- Read model (витрины/queries): денорм, предрасчёты, агрегаты, search-индексы.
