# MIGRATIONS.md — миграции: порядок, правила, safety

# Контекст
Документ задает правила безопасных миграций схемы и данных без простоя. Используется при планировании изменений БД, раскатке и контроле рисков.


## Правила

- Все изменения схемы делаются только миграциями. Ручные правки в проде = “я люблю ад”.
- Миграции только вперёд. Rollback — через новую миграцию, а не “давайте откатим файл”.
- Каждая миграция атомарна и минимальна. Один смысл — одна миграция. Не “заодно ещё 12 таблиц”.
- Миграции обязаны быть безопасны для zero-downtime. База меняется без остановки сервиса.
- Write и read модели мигрируются отдельно. Не мешаем бухгалтерию (write) и витрину (read).
- Основной паттерн: Expand → Deploy → Migrate Data → Contract
  - Expand: добавляем новое, не ломая старое
    (новая колонка nullable, новая таблица, новый индекс).
  - Deploy: выкатываем код, который умеет жить с обеими версиями схемы.
  - Migrate Data: бэкфилл/пересчёт/переезд данных.
  - Contract: удаляем старое, ставим NOT NULL, удаляем старые колонки/индексы.

## Запрещённые операции (без спец-процедуры)
- ALTER COLUMN TYPE на большой таблице без плана (может заблокировать всё).
- ADD COLUMN ... NOT NULL без default/backfill (сломает запись/заблокирует).
- Создание индекса без CONCURRENTLY в Postgres на больших данных.
- Длинные транзакции миграций. Если миграция “думает” минуты — это авария в расписании.

## Индексы: правила
- Индексы на больших таблицах создаются CONCURRENTLY.
- Удаление индексов — тоже аккуратно. Сначала убедись, что не используется (планы/метрики), потом убирай.
- Каждый индекс в миграции должен быть описан в INDEXES.md. Иначе это мусор без владельца.

## Backfill / перерасчёты
- Backfill не делается внутри DDL-миграции, если это долго. Делается отдельным job/скриптом, кусками (batch).
- Backfill должен быть идемпотентным. Можно запустить повторно — будет ок.
- Backfill должен быть наблюдаемым. Логи, прогресс, метрики, безопасная остановка/продолжение.
- Никаких full-table update без батчей. Иначе ты устроишь “блокировки и слёзы”.

## Версионирование и совместимость
- Код должен быть совместим с двумя версиями схемы на период rollout.
- Контракт на чтение/запись фиксируется: какие поля обязательны, какие временно nullable.
- События/outbox мигрируются тоже по expand/contract. Не ломаем потребителей.

## Порядок выполнения (pipeline)
- Dev → Test → Stage → Prod. Никаких “прямо в прод, я уверен”.
- Миграции накатываются до выката кода, когда это безопасно (expand).
- Contract делается после того, как весь прод на новом коде.
- Каждая миграция имеет checksum/версию и применяется один раз.

## Стиль и дисциплина
- DDL и DML разделяем. Схема отдельно, данные отдельно (если долго).
- Feature flags приветствуются. Особенно при изменениях поведения + схемы.
- Удаление колонок — последним шагом. И только после “мы точно не читаем”.
- Миграции документируются. Короткий текст: зачем, как, риски.
