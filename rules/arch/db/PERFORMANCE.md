# PERFORMANCE.md — правила производительности

# Контекст
Документ определяет принципы производительности для write/read путей, включая бюджеты, наблюдаемость и тестирование. Используется для контроля регрессий.


## Правила

## Общие принципы
- Производительность — часть контракта системы.
- “Потом оптимизируем” = “потом перепишем”.
- Write и Read пути оптимизируются отдельно.
- У них разные цели и разные бюджеты.
- Любая оптимизация имеет измеримую цель.
- Без метрик оптимизации не существует.

## Бюджеты производительности
- Для каждого query определяется бюджет.
- Минимум фиксируется:
  - latency (p50 / p95 / p99)
  - максимальный объём данных
  - ожидаемая частота
- Бюджеты документируются и проверяются.
- Нарушение бюджета = дефект.

## Write-путь
- Write-операции минимальны.
- Пишем только то, что меняется.
- Индексы на write-пути ограничены.
- Каждый индекс — это налог на запись.
- Write не зависит от read-моделей.
- Запись не ждёт проекций, кешей и индексов поиска.
- Транзакции короткие.
- Долгие транзакции — враг пропускной способности.

## Read-путь
- Read-модели проектируются под конкретные запросы.
- Один экран ≠ один универсальный SELECT.
- Пагинация обязательна.
- LIMIT/OFFSET или cursor-based — без исключений.
- N+1 запрещён.
- Обнаружение N+1 — автоматическая задача на фикс.
- Используются индексы, а не надежда.
- Каждый тяжёлый query имеет EXPLAIN.
- Сортировки и фильтры ограничены контрактом.
- Нет произвольных ORDER BY user_input.

## Кеширование
- Кеш — часть read-слоя.
- Не хаотичный, а спроектированный.
- TTL и стратегия инвалидции обязательны.
- “Почистим руками” — не стратегия.
- Кеш не является источником истины.
- Промах кеша не ломает функциональность.

## Проекции и асинхронность
- Материализованные read-модели обновляются асинхронно.
- Eventual consistency задокументирована.
- Лаг проекций измеряется.
- Нет метрики — нет гарантии.

## Масштабирование
- Вертикаль прежде горизонтали.
- Сначала индексы и запросы, потом шардинг.
- Шардинг — доменное решение.
- По tenant/aggregate, а не “потому что выросло”.
- Read replicas используются осознанно.
- С учётом лагов и консистентности.

## Наблюдаемость
- Метрики обязательны:
  - latency write/read
  - количество и длительность транзакций
  - количество конфликтов версий
  - медленные запросы
- Slow queries логируются.
- С контекстом, а не “SELECT *”.
- Регресс производительности блокирует релиз.

## Тестирование производительности
- Нагрузочные тесты повторяемы.
- Один сценарий → один результат.
- Тесты отражают реальные query-паттерны.
- Не синтетический ад.
- Тесты входят в CI для критичных кейсов.

## Антипаттерны (запрещено)
- Оптимизация без метрик
- SELECT * в проде
- Отсутствие лимитов
- Индексы “на всякий случай”
- Масштабирование вместо исправления запросов

## Мини-шаблон фиксации решения
- Operation: <query/write>
- Budget: <p95 latency>
- Frequency: <rps>
- Indexes: <list>
- Cache: <yes/no + TTL>
- Notes: <trade-offs>
