# VERSIONING.md — правила версионирования

# Контекст
Документ описывает правила версионирования агрегатов, схемы БД и событий. Используется для обеспечения совместимости и контроля конфликтов.


## Правила

## Версионирование агрегатов и данных
- Каждый агрегат имеет версию.
- version / row_version хранится в root-таблице агрегата.
- Версия используется для optimistic locking.
- Несовпадение версии при UPDATE → конфликт, не “последний победил”.
- Версия инкрементируется при каждом успешном изменении агрегата.
- Чтение версию не меняет.
- Проверка версии выполняется в write-операции.
- Не “где-то потом”, а в момент коммита.
- Конфликты версий обрабатываются явно.
- Retry с лимитом или доменная ошибка — по сценарию.

## Версионирование схемы БД
- Схема БД версионируется миграциями.
- Каждое изменение — отдельная миграция, порядок фиксирован.
- Миграции необратимы логически.
- Rollback — новый forward-migration, не DOWN.
- Изменения схемы делаются по принципу expand/contract.
- Сначала добавляем, потом используем, потом удаляем.
- Breaking changes требуют совместимости.
- Старый код и новая схема должны жить одновременно в переходный период.

## Версионирование событий
- Каждое событие имеет event_version.
- Изменение payload → новая версия события.
- Старые версии событий продолжают обрабатываться.
- Ломающие изменения без адаптера запрещены.
- Версия события не равна версии агрегата.
- Это разные оси времени, не путать.

## Версионирование read-моделей и API
- Read DTO и API версионируются контрактно.
- Изменение полей → новая версия (v2, v3).
- Удаление поля — breaking change.
- Допустимо только через новую версию.

## Антипаттерны (запрещено)
- Отсутствие версии у агрегата
- last write wins без фиксации конфликтов
- Перезапись событий “чтоб исправить”
- Миграции, ломающие прод без переходного периода

## NAMING.md — правила нейминга

### Общие принципы
- Нейминг отражает ubiquitous language.
- Как говорят в домене — так называется код и БД.
- Один термин — одно значение.
- Синонимы запрещены.
- Нейминг стабилен.
- Переименование — изменение контракта, требует миграции/версии.

### Таблицы
- Имена таблиц — в единственном числе.
- agent, order, invoice, не agents.
- Root-таблица агрегата называется по агрегату.
- Без префиксов tbl_, data_.
- Child-таблицы: <root>_<entity>.
- order_item, agent_skill.
- Read-модели имеют суффикс _view или _read.
- order_summary_view.

### Поля
- PK всегда id.
- Тип единообразен в рамках контекста.
- FK всегда <entity>_id.
- order_id, user_id.
- Версия агрегата — version.
- Без фантазий.
- Timestamps:
  - created_at
  - updated_at
  - deleted_at (если soft delete)
- Boolean — is_* / has_*.
- is_active, has_access.

### Индексы и constraints
- Индекс: idx_<table>_<fields>
- Unique: ux_<table>_<fields>
- FK: fk_<from>_<to>
- Check: ck_<table>_<rule>

### События
- Past tense, PascalCase.
- AgentPublished, InvoicePaid.
- Без технических слов.
- Не Event, не Message, не CreatedSuccessfully.

### Команды
- Imperative, PascalCase.
- PublishAgent, PayInvoice.

### Queries / Views
- Описывают результат, не действие.
- AgentListView, InvoiceSummary.

### Антипаттерны (запрещено)
- tbl_, data_, new_, temp_
- foo, bar, test1
- Разные названия для одной сущности
- Переименование “потому что так красивее”
