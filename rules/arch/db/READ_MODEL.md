# READ_MODEL.md — правила витрин/проекций (queries, денорм)

# Контекст
Документ описывает правила проектирования read-моделей, витрин и проекций. Фиксирует требования к обновлению, консистентности и контрактам чтения.


## Правила

- Read-модель — только для чтения. Никаких INSERT/UPDATE/DELETE бизнес-состояния через витрины.
- Read-модель не обязана совпадать с доменной моделью. Это не “истина”, это “быстро показать”.
- Read-модель строится от query-кейсов. Сначала список экранов/отчётов/эндпоинтов, потом схема.
- Read-модель может агрегировать данные из нескольких агрегатов. Это разрешено, потому что она не транзакционная граница.
- Read-модель денормализуется по умолчанию. Дублировать данные для скорости — нормально, если есть источник истины в write.
- Read-модель не содержит бизнес-инвариантов. Инварианты живут в write-модели/агрегатах, здесь — отображение.
- Queries не грузят агрегаты. Репозитории агрегатов не используются для чтения “потому что так привычно”.
- Каждый query имеет явную проекцию/DTO. Не возвращаем доменные сущности наружу.
- Read-модель проектируется под предсказуемую производительность. Пагинация обязательна, лимиты обязательны, N+1 запрещён.
- Индексы в read-модели делаются под реальные запросы. Каждая витрина → список запросов → EXPLAIN → индексы.
- Read-модель может обновляться:
  - on-demand (прямой запрос к write БД) — если нагрузка/сложность ок
  - materialized (проекции по событиям) — если нужен быстрый UI/поиск/аналитика
- Если read-модель материализованная — источник данных только события/outbox. Не “давай джойним напрямую и обновляем по крону”.
- Обновление проекций идемпотентно. Повтор события/доставка дважды не ломает витрину.
- Порядок обработки гарантируется только внутри одного агрегата. Между агрегатами — eventual consistency, и это документируется.
- Консистентность read-модели формализуется. Для каждой витрины: freshness SLA (например, “до 5 секунд”).
- Read-модель версионируется на уровне контрактов. DTO/эндпоинты имеют версию при изменениях схемы.
- Ошибки read-слоя не влияют на write. Падает витрина — запись работает, иначе ты построил монолит-ловушку.
- Кеширование — часть read-слоя. Явный TTL/инвалидация, запрет “кеш навсегда, авось пронесёт”.
- Поиск/фильтры/сортировки описываются контрактно. Какие поля доступны, какие индексы обеспечивают.
- Безопасность и доступы применяются на чтении отдельно. RBAC/tenant-фильтры — не “в UI спрячем”.
- PII в read-моделях минимизируется. Если надо — маскирование/шифрование/ограничение доступа.
- Read-модель не является источником истины. Любой спор → идём в write-модель.
