# INDEXES.md — индексы, планы, запреты на N+1

# Контекст
Документ описывает подход к индексации и валидации планов запросов. Регулирует, какие индексы допустимы для write/read путей и как подтверждать их пользу.


## Правила

- Индексы ставятся под реальные запросы. Не “на всякий”, а под конкретные query-паттерны.
- Каждый индекс обязан иметь владельца: список запросов + ожидаемый профит. “Красиво” не считается.
- Write-модель: индексы минимальные. Только для:
  - PK/FK
  - UNIQUE constraints
  - идемпотентности
  - hot-path обновлений/поиска по ключу
- Read-модель: индексы агрессивные. Там читаем быстро, пишем не так жалко.
- Любой новый индекс → проверка EXPLAIN (ANALYZE, BUFFERS). Без плана — индекс не существует.
- Каждый индекс должен быть селективным. Индекс по полю, где 90% значений одинаковые — это мусор.
- Составные индексы строятся по порядку фильтров. Сначала equality (=), потом range (> < BETWEEN), потом сортировка (ORDER BY).
- Индексы под сортировку и пагинацию обязательны. Иначе “страница 500” станет DDOS’ом самому себе.
- Partial indexes используются осознанно. Например, по status='ACTIVE' — если это основной фильтр.
- Covering indexes (INCLUDE) — для read-хотпатов. Чтобы закрывать запрос без похода в heap.
- GIN/Full-text индексы только когда нужен поиск. Не заменяют нормальные фильтры.
- Индексы должны быть совместимы с мульти-тенантностью. Часто ключ = (tenant_id, …).

## Запреты (жёстко)
- N+1 запрещён. Любой код, который делает “получили список → на каждый элемент отдельный запрос” — в мусор.
- JOIN “на всё подряд” запрещён. Если запрос тащит полмира ради одной страницы — делай read-модель/витрину.
- OFFSET без стратегии запрещён на больших данных. Используй keyset pagination (по курсору), а не “листать вселенную”.

## Процесс добавления/изменения индексов
- Шаг 1: зафиксировать запрос. SQL + параметры + ожидаемый объём данных.
- Шаг 2: снять план. EXPLAIN (ANALYZE, BUFFERS) до индекса.
- Шаг 3: добавить индекс. В миграции, безопасно (в Postgres — CONCURRENTLY).
- Шаг 4: снять план после. Сравнить latency/IO.
- Шаг 5: проверить стоимость записи. Индекс ускоряет чтение, но может убить insert/update.
- Шаг 6: добавить в каталог индексов. Чтобы не плодить дубликаты.

## Стандарты индексации (полезные дефолты)
- PK индекс — всегда.
- FK колонки индексируются. Иначе join и каскады внутри агрегата будут страдать.
- (tenant_id, business_key) — индекс обязателен при мульти-тенантности.
- created_at индекс — если есть фильтры/сортировка по времени.
- status + updated_at — если есть “очереди” и выборки по статусам.

## Отдельно: правила против N+1
- Списки грузим одним запросом. JOIN/IN/CTE/витрина — что уместно.
- Если нужен “сложный UI список” — делаем read-модель. Не мучаем write-схему.
- ORM lazy-loading в чтении запрещён по умолчанию. Разрешается только там, где гарантирован один объект, а не список.
