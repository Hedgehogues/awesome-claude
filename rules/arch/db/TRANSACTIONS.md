# TRANSACTIONS.md — правила транзакций

# Контекст
Документ задает правила транзакций, границы агрегатов и обязательный outbox для событий в write-модели. Определяет, где открывается транзакция, какие операции допустимы внутри и как обеспечивается идемпотентность и консистентность.

## Правила

- Транзакция ограничена одним агрегатом.
- Любая попытка писать два агрегата в одной транзакции — архитектурное нарушение.
- Агрегат — атомарная единица записи.
- Либо весь агрегат закоммичен, либо весь откатан. Частичных состояний не существует.
- Транзакция открывается на уровне application service / command handler.
- Не в контроллере, не в агрегате, не “где удобнее”.
- Внутри транзакции разрешены только операции одного репозитория.
- Один репозиторий ↔ один агрегат ↔ одна транзакция.
- Вложенные транзакции запрещены.
- REQUIRES_NEW, savepoint’ы и прочие трюки — только с явным обоснованием и документированием.
- Транзакции минимальны по времени жизни.
- Никаких сетевых вызовов, очередей, HTTP, sleep и ожиданий внутри транзакции.
- Изоляция выбирается осознанно и фиксируется.
- По умолчанию:
  - READ COMMITTED — стандарт
  - REPEATABLE READ — если нужен стабильный срез
  - SERIALIZABLE — только при строгой необходимости
- Уровень изоляции не используется для бизнес-логики.
- Инварианты держатся агрегатом и constraints’ами, не “магией БД”.
- Optimistic locking — основной механизм конкуренции.
- version проверяется при записи. Конфликт → retry или доменная ошибка.
- Pessimistic locking используется редко и локально.
- SELECT ... FOR UPDATE — только при доказанной гонке и под конкретный кейс.
- Любая транзакция может быть прервана БД.
- Deadlock, serialization failure — это норма, код обязан уметь retry.
- Retry выполняется на уровне application service.
- С backoff и лимитом попыток. Бесконечных ретраев не существует.
- Идемпотентность обязательна для внешних входов.
- Повтор команды не должен приводить к повторному эффекту.
- Outbox пишется в той же транзакции, что и агрегат.
- События без этого — потенциальная ложь.
- Побочные эффекты запрещены внутри транзакции.
- Email, HTTP, Kafka, Telegram — строго после коммита.
- Чтение внутри write-транзакции допускается только для текущего агрегата.
- Cross-aggregate reads для “проверок” — запрещены.
- Read-модель не обновляется в write-транзакции.
- Проекции строятся асинхронно.
- Тайм-аут транзакции ограничен.
- Длинные транзакции — дефект, а не “сложный кейс”.
- Логирование и трассировка обязательны.
- transaction_id, command_id, aggregate_id, retry_count.

### Типовые паттерны
Create aggregate:
- начать транзакцию
- сохранить root + children
- записать outbox
- commit

Update aggregate:
- загрузить агрегат
- применить команду
- проверить version
- сохранить
- outbox
- commit

Conflict:
- rollback
- retry или вернуть доменную ошибку

### Антипаттерны (запрещено)
- Одна транзакция на несколько агрегатов
- Транзакция + HTTP/очередь внутри
- “Проверим другой агрегат и потом запишем”
- SERIALIZABLE как костыль
- Savepoint’ы вместо нормальной модели

### OUTBOX.md — правила публикации событий
- Outbox — обязательный механизм доставки событий.
- Прямая публикация из кода без outbox запрещена.
- События записываются в outbox в одной транзакции с агрегатом.
- Если агрегат закоммичен — событие обязано существовать. И наоборот.
- Outbox — часть write-модели.
- Это не лог, не очередь, не “временное решение”.
- Каждая запись outbox соответствует одному событию.
- Никаких батчей, массивов и “оптимизаций”.
- Outbox не содержит бизнес-логики.
- Только данные события + метаданные доставки.

#### Структура outbox-события (минимальный контракт)
- event_id — глобально уникальный идентификатор
- event_type — имя события (домена/интеграции)
- event_version — версия схемы события
- aggregate_id — источник события
- aggregate_type — тип агрегата
- aggregate_version — версия агрегата на момент события
- payload — данные события (JSON)
- occurred_at — время факта
- correlation_id — сквозная трассировка
- causation_id — что породило событие
- status — NEW | SENT | FAILED
- retry_count — количество попыток доставки

#### Правила записи
- Outbox пишется только после успешного применения команды.
- Событие отражает факт, а не намерение.
- Порядок событий гарантируется только в рамках одного агрегата.
- Между агрегатами — без гарантий.
- Payload минимальный.
- ID и необходимые поля. Никаких графов объектов.
- События неизменяемы.
- Ошибка → новое событие, не UPDATE payload.

#### Доставка событий
- Публикация выполняется отдельным воркером/процессом.
- Application service этим не занимается.
- Чтение из outbox — по статусу и/или watermark.
- Без full scan таблицы.
- Доставка идемпотентна.
- Повторная отправка одного события допустима и безопасна.
- После успешной доставки событие помечается как SENT.
- Удаление — отдельной политикой (см. RETENTION).
- Ошибки доставки не блокируют write-путь.
- Write всегда важнее read/интеграций.

### Повторы и ошибки
- Retry обязателен.
- С backoff и лимитом попыток.
- Poison events не блокируют очередь.
- После лимита → FAILED + алерт.
- Обработчики обязаны быть идемпотентными.
- event_id используется для дедупликации.

### Версионирование
- События версионируются явно.
- Новая схема → новая версия, не “сломаем потребителей”.
- Outbox хранит события всех версий.
Маппинг/адаптация — на стороне потребителя.

Антипаттерны (запрещено)

Публиковать события до коммита агрегата

Писать в Kafka/Rabbit/HTTP из транзакции

Делать outbox “в памяти”

Менять payload уже записанного события

Использовать outbox как event store
