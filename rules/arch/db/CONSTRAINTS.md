# CONSTRAINTS.md — ограничения БД (FK/UNIQUE/CHECK) и инварианты

# Контекст
Документ фиксирует правила по ограничениям БД (PK/FK/UNIQUE/CHECK/NOT NULL), которые защищают бизнес-инварианты и ключи в write-модели. Используется при проектировании схем и проверке соблюдения целостности.


## Правила

- Constraints обязательны. Если инвариант можно зафиксировать в БД — он фиксируется. “В коде проверим” не считается.
- Каждый constraint должен быть привязан к бизнес-правилу. В документе: что защищает и почему это важно.
- Write-модель защищается constraints’ами, read-модель — по необходимости. В read обычно достаточно индексов, не “железных правил”.
- Внутри агрегата — строгие FK и каскады допустимы. Между агрегатами — только ссылки по ID, каскады запрещены.
- UNIQUE — основной механизм защиты business keys. Email/slug/номер документа/идемпотентность — всё через UNIQUE.
- CHECK — для простых доменных условий. Диапазоны, перечисления, форматы, “amount > 0”, “status in (…)”.
- NOT NULL — по умолчанию для обязательных полей. Nullable — только если есть сценарий “нет значения” в домене.
- DELETE/UPDATE CASCADE запрещены между агрегатами. Иначе ты устроишь “удалил пользователя — удалил вселенную”.
- DEFERRABLE constraints используются осознанно. Только если реально нужно отложить проверку до commit.
- Constraints не заменяют доменную логику. Они не “умные”, они “страховка”. Инварианты уровня поведения остаются в агрегате.
- Идемпотентность должна быть защищена constraints’ами. UNIQUE(idempotency_key) или UNIQUE(external_event_id).
- Мульти-тенантность учитывается во всех UNIQUE. Обычно UNIQUE(tenant_id, business_key) вместо глобальных уникальностей.
- Ссылочная целостность должна быть определена явно. Если FK нет — причина документируется (шардинг, асинхронность, внешние системы).
- Каждый constraint имеет стандартное имя. Чтобы не было “uq_12345_abcdef”.

## Политики по типам constraints

### Primary Key
- PK обязателен в каждой write-таблице.
- PK неизменяем. Никаких “переехали на другой id”.

### Foreign Key (FK)
FK внутри агрегата:
- обязателен, если зависимость строгая
- ON DELETE CASCADE допустим для owned-коллекций

FK между агрегатами:
- допускается только как “ссылка по ID”
- ON DELETE/UPDATE только RESTRICT/NO ACTION
- без каскадов

### UNIQUE
- UNIQUE на business keys обязателен.
- UNIQUE на технические ключи идемпотентности обязателен.
- Составные UNIQUE для тенанта/домена обязательны.

### CHECK
CHECK для простых правил:
- amount >= 0
- status IN (...)
- starts_at < ends_at

Сложные правила не пихать в CHECK, если это ведёт к монстру. Они живут в домене.

### NOT NULL
- NOT NULL по умолчанию.
- Nullable только если “отсутствие значения” — валидное состояние.

## Шаблон описания constraint’ов (в документе)
Для каждой таблицы:
- Table: <name>
- Constraints:
- PK: <pk_name>(id) — идентичность записи
- FK: <fk_name>(child_id → parent.id) — владение внутри агрегата
- UNIQUE: <uq_name>(tenant_id, slug) — slug уникален в тенанте
- CHECK: <ck_name>(amount > 0) — сумма всегда положительная
- NOT NULL: <column> — обязательное поле

## Стандарты именования (пример)
- pk_<table>
- fk_<from_table>__<to_table>__<column>
- uq_<table>__<columns>
- ck_<table>__<rule_short>

## Красные флаги (если увидел — заводи техдолг)
- “UNIQUE в коде” вместо UNIQUE в БД
- каскады между агрегатами
- nullable поля без доменного смысла
- отсутствие constraints на business keys
- JSON вместо нормальных полей для ключевых правил
