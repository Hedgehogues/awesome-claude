# RETENTION.md — правила хранения и удаления данных

# Контекст
Документ задает политику хранения, удаления и архивирования данных. Используется для соблюдения доменных и юридических требований по срокам хранения.


## Правила

## Общие принципы
- Хранение данных — осознанное решение.
- “Оставим на всякий случай” — не стратегия, а будущий инцидент.
- Каждый тип данных имеет политику ретенции.
- Если нет политики — данные считаются временными.
- Источник истины важнее истории.
- История хранится только если она нужна домену, аудиту или закону.

## Классификация данных
- Операционные данные (write-модель).
  - Актуальное состояние агрегатов.
- Исторические данные.
  - События, аудит, изменения состояний.
- Read-модели / витрины.
  - Производные данные, допускающие пересборку.
- Технические данные.
  - Логи, метрики, служебные таблицы.

## Политики хранения
- Write-модель хранится бессрочно, если иное не задано доменом.
- Удаление — только как бизнес-операция.
- Soft delete применяется только по доменному требованию.
- deleted_at — это состояние, а не “чтоб не удалять”.
- Hard delete применяется по истечении ретенции.
- После архивации или если данные восстанавливаемы.
- Read-модели считаются восстановимыми.
- Их можно удалять и пересобирать без потери истины.
- Outbox и события имеют ограниченный TTL.
- После доставки и периода гарантии — архив или удаление.

## PII и чувствительные данные
- PII удаляется по минимально допустимому сроку.
- Закон > бизнес > удобство.
- Удаление PII должно быть полным.
- Не только write-модель, но и:
  - read-модели
  - кэши
  - индексы поиска
  - бэкапы (по политике)
- Анонимизация предпочтительнее физического удаления, если допустимо.
- Особенно для аналитики.

## Архивация
- Архив — не прод.
- Отдельное хранилище, другой SLA, другие доступы.
- Архивируем данные, которые:
  - редко читаются
  - нужны для аудита/аналитики
  - не участвуют в бизнес-инвариантах
- Архивируем в формате, пригодном для восстановления/анализа.
- Не “дамп и забыли”.

## Outbox, события, логи
- Outbox:
  - NEW/FAILED — храним до разруливания
  - SENT — TTL ограничен
- Доменные события не являются event store.
- Хранение ограничено ретенцией.
- Технические логи имеют короткий TTL.
- Отладка ≠ архив.

## Процедуры удаления
- Удаление — асинхронная операция.
- Не блокирует write-путь.
- Удаление идемпотентно.
- Повтор не ломает состояние.
- Удаление логируется.
- Кто, что, когда, по какой причине.
- Удаление тестируется.
- Ретенция без тестов = миф.

## Контроль и аудит
- Политики ретенции документированы и проверяемы.
- Не “в голове у тимлида”.
- Регулярные проверки соблюдения.
- Джобы/алерты на превышение TTL.
- Нарушение ретенции — инцидент.
- Это риск, не “потом подчистим”.

## Антипаттерны (запрещено)
- Хранить данные “на всякий случай”
- Soft delete без политики hard delete
- Дубли PII в read-моделях без TTL
- Архив в той же БД и с теми же доступами
- Ручное удаление из прод-БД
