# Visual Cohesion & Coupling

## Правило

UI-области, оперирующие одним агрегатом и выполняющие одну доменную операцию,
ДОЛЖНЫ использовать один layout-паттерн, один CSS-класс и одну ось выравнивания.

Разный визуал допустим только при пересечении границы bounded context или при
разной доменной операции.

## Cohesion (группировка)

| Проверка | Вопрос |
|---|---|
| **Pattern cohesion** | Компоненты с одинаковой функцией используют один CSS-класс? |
| **Axis cohesion** | Элементы одного типа выровнены по общей вертикальной/горизонтальной оси? |
| **Layout cohesion** | Все элементы внутри компонента участвуют в одном layout-паттерне? |

**Как определять**: если два UI-блока принадлежат одному агрегату и выполняют
одну операцию (например, "контент + комментарии к части документа") — это один
визуальный паттерн. Один grid, один набор CSS-переменных.

## Coupling (связанность)

| Проверка | Вопрос |
|---|---|
| **DRY** | Layout-значение (ширина колонки, gap, padding) определено в одном месте? |
| **Token coupling** | Используются CSS-переменные или хардкод? |
| **Override coupling** | Есть ли CSS-override, дублирующий значение из базового класса? |

**Правило**: единый паттерн определяется в одном месте (один класс, одна переменная).
Если значение продублировано — изменение в одном месте не дойдёт до другого,
и cohesion разрушится при следующей правке.

## Примеры

### Нарушение (low cohesion + implicit coupling)

```css
/* Description row — свой grid */
.document-prompt { grid-template-columns: 1fr 340px; }

/* Block row — другой grid, та же 340px */
.block-editor-row { grid-template-columns: 1fr 4px 340px; }
```

Один паттерн (контент + комменты), два layout-а, значение `340px` захардкожено
в двух местах.

### Правильно (high cohesion + low coupling)

```css
:root { --comments-width: 340px; }

.content-row {
  display: grid;
  grid-template-columns: 1fr var(--comments-width);
}
```

И Description, и Block используют `.content-row`. Меняешь переменную — меняется
везде.

## Когда разный layout допустим

- Разные bounded context (Blocks vs Event Storming)
- Разные доменные операции (редактирование vs просмотр истории)
- Разные viewport breakpoints (мобильный vs десктоп)
