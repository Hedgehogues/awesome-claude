# LOGS.md — логгирование

## Контекст
Логгирование является частью архитектуры и не рассматривается как отладочный механизм.
Логи не являются источником истины; источником истины выступают агрегаты, события и база данных.
Логи единообразны по всей системе и следуют единому формату и контракту.

## Правила
### Формат логов
- Только структурированные логи. JSON или эквивалент.
- Каждая запись содержит обязательные поля:
  - timestamp
  - level
  - service
  - environment
  - message
  - correlation_id
- Контекстные поля добавляются по необходимости:
  - command_id
  - request_id
  - aggregate_id
  - aggregate_type
  - event_id
  - tenant_id

### Уровни логирования
- ERROR — операция не завершена, требуется реакция.
- WARN — отклонение от нормы, система продолжает работу.
- INFO — значимые бизнес-переходы.
- DEBUG — диагностическая информация, не включена постоянно.
- TRACE — низкоуровневый шум, только локально.
- Уровни не используются для сокрытия проблем.
- Если информация критична для бизнеса, она фиксируется на уровне INFO, а не DEBUG.

### Логгирование write-пути (команды)
- Каждая команда логируется на входе и выходе.
- Фиксируется:
  - тип команды
  - идентификатор агрегата
  - версия агрегата (до/после)
  - результат (успех/отказ)
- Payload команды полностью не логируется.
- Доменные ошибки логируются как ожидаемые.

### Логгирование read-пути (queries)
- Read-запросы логируются агрегированно.
- Фиксируется:
  - тип запроса
  - параметры пагинации
  - количество результатов
  - latency
- PII в read-логах запрещён.

### Логгирование событий
- Факт публикации события логируется.
- Фиксируется:
  - event_id
  - event_type
  - aggregate_id
- Payload события не логируется.
- Ошибки доставки логируются отдельно от обработки.

### Ошибки и исключения
- Ошибки типизированы.
- Логи содержат error_code, а не свободный текст.
- Stack trace пишется только при ERROR.
- Ошибки доступа и безопасности логируются без деталей.

### Корреляция и трассировка
- Correlation ID обязателен и должен быть сквозным.
- Отсутствие correlation_id является архитектурным дефектом.
- Correlation ID передаётся через команды, события и асинхронные процессы.

### Безопасность логов
- PII запрещён в логах.
- Секреты запрещены в логах.
- Логи считаются чувствительными данными.
- Доступ к логам ограничен по ролям.

### Хранение и ретенция логов
- Логи хранятся ограниченное время.
- Разная ретенция по уровням:
  - ERROR/WARN — дольше
  - INFO — ограниченно
  - DEBUG/TRACE — минимально
- Логи не хранятся вечно.

### Алерты
- Алерты строятся по сигналам, не по шуму.
- Алерты основаны на:
  - росте ERROR
  - росте latency
  - повторяющихся доменных отказах
- Алерты по уровням DEBUG/TRACE не допускаются.

## Запрещено (жёстко)
- Логировать payload целиком.
- Логировать PII и секреты.
- Использовать логи как event store.
- Вести «отладочные» логи в продуктивной среде.
- Отсутствие корреляции.

## Минимальный эталон лог-записи
```json
{
  "timestamp": "2026-01-12T03:17:00Z",
  "level": "INFO",
  "service": "billing",
  "message": "command_succeeded",
  "command": "PayInvoice",
  "aggregate_id": "invoice-123",
  "version": 5,
  "correlation_id": "abc-xyz"
}
```
