# UI Component Library

Библиотека компонентов строится **сверху вниз** — от реальных повторяющихся паттернов на экране к абстрактным примитивам. Главная цель: убрать дублирование между страницами и дать единый визуальный язык.

## Архитектура: четыре слоя

```
components/
├── ui/                    # Слой 1–2: tokens + примитивы
├── shared/                # Слой 3: пользовательские компоненты (видимые паттерны)
├── domain/                # Слой 4: бизнес-компоненты (знают о типах и API)
└── layout/                # App shell
```

### Слой 1: Design Tokens (`ui/tokens/`)

CSS-переменные — единственный источник визуальных констант (цвета, отступы, типографика, радиусы, тени, переходы). Тема переключается через `[data-theme]`.

- Семантические имена (`--bg-card`), а не абстрактные (`--gray-800`)
- Жёстко прописанные значения в компонентах запрещены — только `var(--token)`
- Новый токен требует обоснования: нельзя ли использовать существующий?

### Слой 2: UI-примитивы (`ui/`)

Атомарные элементы без доменных зависимостей: `Button`, `IconButton`, `Input`, `Checkbox`, `Badge`, `Card`, `Dialog`, `Spinner`, `Pagination`, `EmptyState` и т.д.

- Никаких импортов доменных типов, API-клиентов, контекстов, роутера
- Вариативность через `variant` / `size` props
- Контролируемое состояние (`value` + `onChange`)
- CSS Modules, ARIA, no side effects

### Слой 3: Пользовательские компоненты (`shared/`)

**Это главный слой библиотеки.** Здесь живут паттерны, которые пользователь реально видит на экране и которые повторяются на 2+ страницах. Каждый компонент собирает примитивы из `ui/` в законченный UI-блок.

#### Принцип: Extract from Duplication

Компонент попадает в `shared/` **только когда** один и тот же визуальный паттерн встречается на 2+ страницах. Не проектировать абстракции заранее — сначала заметить дубль, потом извлечь.

**Антипаттерн:** создавать `shared/` компонент для паттерна, который встречается только в одном месте. Такой компонент остаётся в `domain/` или в странице.

#### Типичные кандидаты для `shared/`

| Паттерн | Признак дублирования |
|---------|---------------------|
| **Карточка с header/body/footer** | Несколько страниц рендерят карточки с одинаковой структурой, но разными CSS-классами |
| **Inline edit** | Переключение display → edit mode с save/cancel копируется в 2+ местах |
| **Нумерованный блок** | Номер + заголовок + статус + actions повторяется на разных страницах |
| **Toolbar с группами действий** | Горизонтальная панель иконок с разделителями и счётчиками |
| **Status dot** | Цветная точка с tooltip как индикатор состояния |
| **Список с create + empty state** | Список элементов с кнопкой создания и пустым состоянием |

#### Правила для `shared/`

- **Не импортируют** доменные типы — принимают generic props
- **Могут содержать** внутреннее UI-состояние (`useState` для editing mode, expanded и т.д.)
- **Не делают** HTTP-запросы — получают данные и callbacks через props
- **Используют** примитивы из `ui/`
- CSS Modules, design tokens, ARIA

#### Отличие от `ui/` примитивов

| | `ui/` (примитивы) | `shared/` (пользовательские) |
|---|---|---|
| Что это | Атомарные элементы (кнопка, поле) | Законченные UI-блоки (карточка, inline-edit) |
| Аналог | HTML-элемент | Виджет |
| Внутреннее состояние | Минимальное | Может быть (editing mode, expanded) |
| Пользователь видит | Косвенно | Напрямую на экране |

### Слой 4: Доменные компоненты (`domain/`)

Знают о бизнес-типах, вызывают API, используют контексты. Собирают `shared/` и `ui/` компоненты, маппя доменные данные в generic props.

## Порядок создания нового `shared/` компонента

1. **Заметить дубль** — один и тот же визуальный паттерн на 2+ страницах
2. **Определить общий интерфейс** — что общего у всех использований, что различается
3. **Написать тесты** — рендер, варианты, состояния, interactions
4. **Реализовать** — `shared/ComponentName/` с CSS Module и примитивами из `ui/`
5. **Заменить дубли** — обновить все места, где паттерн был inline
6. **Удалить дублирующийся CSS** из глобальных стилей
7. `make check` — зелёный

## Инженерные требования (общие для `ui/` и `shared/`)

### Стилизация
- CSS Modules (`.module.css`) — изоляция имён классов
- Каждый компонент — директория: `ComponentName/ComponentName.tsx` + `.module.css` + `.test.tsx`
- Все значения — через design tokens
- Inline styles только для динамических значений

### Props API
- `variant` / `size` для визуальных вариантов
- Контролируемое состояние (`value` + `onChange`)
- `children` для композиции
- `className` на корневой элемент для точечных переопределений
- Callbacks без side effects внутри компонента

### Доступность
- ARIA-атрибуты на интерактивных элементах
- Keyboard navigation (Enter/Space/Escape)
- `aria-label` + `title` на icon-only кнопках

### Контракт изменений
- Изменение props API = breaking change → обновить всех потребителей
- Добавление нового variant = не breaking
- Каждое изменение — отдельный коммит, `make check` после каждого
