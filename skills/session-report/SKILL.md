# Session Report Skill

Сгенерируй продуктовый отчёт по текущим незакоммиченным изменениям.

## Порядок выполнения

### Шаг 1: Собери данные

Выполни параллельно:

```bash
# Список изменённых файлов с количеством строк
git diff --stat HEAD

# Список untracked файлов (новые)
git status --porcelain | grep '^??' | sed 's/^?? //'

# Последние коммиты для контекста
git log --oneline -5
```

### Шаг 2: Определи продуктовые изменения

Используй Agent (subagent_type=Explore) чтобы прочитать ключевые изменённые файлы.

Фокусируйся на **пользовательских фичах**, а не на инфраструктуре:

**Читай в первую очередь:**
- `presentation/api/routes.py` — новые API endpoints
- `presentation/api/schemas.py` — новые request/response схемы
- `domain/**/entity.py` — новые доменные методы и поля
- `pages/*.tsx`, `components/**/*.tsx` — новые UI компоненты и страницы
- `*.css` — значительные изменения стилей
- Миграции (`migrations/versions/`) — новые таблицы и поля

**Игнорируй (не включай в отчёт):**
- Тесты (упоминай только количество: "покрыто N тестами")
- Рефакторинг без изменения поведения
- Правки линтера, форматирования
- Изменения в CI/CD, Docker, skills, правилах

### Шаг 3: Сгруппируй по продуктовым темам

Объедини связанные изменения в **фичи**. Каждая фича = backend + frontend + миграция.

Для каждой фичи укажи:
- **Название** — что пользователь теперь может делать
- **Суть** — 1-2 предложения, как это работает
- **Масштаб** — сколько файлов затронуто, есть ли миграция

### Шаг 4: Сформируй отчёт

## Формат отчёта

```markdown
## Продуктовый отчёт: [дата]

**Файлов изменено:** N | **Строк:** +X / -Y | **Новых файлов:** M

---

### [Главная тема сессии — 1 предложение]

#### 1. [Название фичи]
[Описание: что может пользователь, как это работает]

#### 2. [Название фичи]
[Описание]

...

---

**Покрытие тестами:** N новых/изменённых тестов
**Миграции:** список новых миграций (если есть)
```

## Режимы

Если `$ARGUMENTS` содержит `short`:
- Только буллеты, без описаний
- Максимум 10 строк

Если `$ARGUMENTS` содержит `full` или пуст:
- Полный формат с описаниями
- Insight-блок в конце с продуктовым выводом

## Правила

- Отвечай на русском
- Не упоминай инфраструктурные изменения (Docker, CI, skills) если они не влияют на пользователя
- Группируй backend + frontend + миграцию в одну фичу, а не перечисляй отдельно
- Называй фичи с точки зрения пользователя ("История версий описания"), а не разработчика ("Добавлена таблица description_versions")
